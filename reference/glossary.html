Rust · Reference

Glossary

The canonical vocabulary for this course. Every lesson uses these terms exactly as defined here. Add to it; don't drift to synonyms.

The ? operator
Postfix on a Result (or Option): unwraps Ok(v) to v, or returns Err(e) early from the enclosing function. Requires that function to return a compatible Result/Option. Lesson 2
assert_eq! / assert!
Test macros: assert_eq!(a, b) panics (failing the test) unless the values are equal, printing both on failure; assert!(cond, "msg") panics unless cond is true. Lesson 3
Borrow (Lesson 4)
Accessing a value through a reference (&x) without taking ownership, so the owner keeps it. The alternative to moving or cloning.
Box<dyn Error> (trait object)
A single type that holds any value implementing the Error trait, with the concrete type resolved at runtime (dyn). Used as a catch-all error type so ? can convert many error kinds into one return type. Lesson 2
Clone
An explicit, potentially expensive deep copy (x.clone()) that duplicates heap data so both bindings own independent values. Rust never does this implicitly. Orientation
Copy (trait)
A marker trait for small, fixed-size, stack-only types (integers, bool, char, f64, tuples of such). Assigning a Copy value duplicates it instead of moving it, so the original stays valid. Orientation
derive (Lesson 6)
An attribute, #[derive(...)], that asks the compiler to generate standard trait implementations automatically — e.g. Debug (the {:?} format), Default (a zero/empty value), Clone/Copy, and PartialEq/Eq (comparison with ==).
Doc-test (Lesson 6)
A runnable example inside a /// doc comment, fenced in triple-backticks. cargo test compiles and runs each one, so documentation can't drift from the code. Collected only for library crates. A leading # hides a line from rendered docs but still runs it; fence words control execution — no_run (compile only), ignore (skip), should_panic (must panic), compile_fail (must not compile).
Drop
The cleanup that runs automatically when an owner goes out of scope (frees heap memory, closes resources). The rough analogue of a C++ destructor; you rarely call it yourself. Orientation
enum (Lesson 6)
A custom type that is exactly one of a fixed set of variants (a "sum type" / tagged union), and a variant may carry data — e.g. Source::Stdin or Source::File(String). The compiler tracks which variant is active; you read it back with match. Option and Result are standard-library enums.
Exhaustiveness (Lesson 7)
A match must cover every possible case. For an enum that means an arm per variant (or a _ wildcard). Add a variant and every non-wildcard match stops compiling (E0004) until it handles the new case — a compile-time guarantee that you never forget one.
Guard (match) (Lesson 7)
An extra condition on a match arm, written pattern if condition. The arm runs only when the pattern matches and the condition is true, e.g. Source::File(path) if path.ends_with(".md"). A guard does not count toward exhaustiveness.
if let / let else (Lesson 7)
Shorthands for matching a single pattern. if let PAT = expr { … } runs the block only when the pattern matches, and the binding is scoped to that block. let PAT = expr else { … } binds when the pattern matches and otherwise runs an else block that must diverge (return/break/continue/panic!); the binding then stays in scope.
Library crate vs binary crate (Lesson 6)
src/main.rs is a binary crate (produces an executable); src/lib.rs is a library crate (reusable code, imported by package name, e.g. use rwc::Counts). A package can hold both, with the binary depending on the library.
match (Lesson 6, 7)
An expression that compares a value against a series of patterns and runs the first arm that fits, producing a value. Works on any type — an enum, an integer, a string slice. Arms can bind a variant's data (Source::File(path)) and carry guards; the set of arms must be exhaustive.
Move
What happens on assignment (or passing by value) of a non-Copy type: ownership transfers to the new binding and the old binding becomes invalid. Using it afterward is a compile error (borrow of moved value). The heap data is not duplicated. Orientation
Option<T>
An enum for a value that may be absent: Some(value) or None. Rust's replacement for null — the compiler forces you to handle the None case before using the value. Lesson 1
Owner
The single variable currently responsible for a value. There is only ever one at a time. Orientation
Ownership
Rust's compile-time discipline for managing memory: every value has exactly one owner, and when the owner goes out of scope the value is freed. No garbage collector, no manual free. Orientation
panic
An unrecoverable abort for bugs/impossible states. Unwinds the stack, prints a message and backtrace hint to stderr, and exits with code 101. Triggered by panic!, .unwrap(), .expect(). Prefer Result for expected failures. Lesson 2
Pattern (Lesson 7)
The shape on the left of a match arm (or in if let, let else, and ordinary let). A pattern can be a literal ("-"), the wildcard (_), a variant that binds its data (Source::File(path)), a range (0..=9), or a binding with @.
Pure function
A function whose output depends only on its inputs, with no I/O or side effects (e.g. rwc's counts(&str)). Trivial to test: pass input, assert on the return value. Lesson 3
Reference: &T / &mut T (Lesson 4)
Temporary access to a value the caller still owns. &T is a shared (read-only) borrow — you may have any number at once; &mut T is a mutable (exclusive) borrow — at most one, and no other borrow may coexist with it. The rule, enforced at compile time: either one &mut or any number of &, never both ("shared XOR mutable"). A reference may never outlive the data it points to.
Result<T, E>
An enum for an operation that may fail: Ok(value) or Err(error). Returned by anything fallible (e.g. file IO). The error must be acknowledged before the success value can be used. Lesson 1, 2
Scope
The region of code (usually a { } block) over which a binding is valid. Ownership and dropping are tied to scope boundaries. Orientation
Stack / Heap
Stack: fast, fixed-size, LIFO storage for values whose size is known at compile time. Heap: storage for data of size unknown or variable at compile time (e.g. a growable String), reached via a pointer held on the stack. Orientation
String vs &str (preview — Module 2)
String is an owned, growable, heap-allocated UTF-8 string; &str is a borrowed view (slice) into UTF-8 text. Detailed in a later lesson.
struct (Lesson 6)
A custom type bundling named fields held together (a "product type") — e.g. Counts { lines, words, chars }. Fields and the type carry pub visibility independently; methods and associated functions live in an impl block. The named fields replace positional tuple access (.0/.1).
Trait (seen: Orientation, Lessons 2 & 6 — full treatment Module 4)
A named set of methods a type can promise to provide — Rust's mechanism for shared behaviour, close to an interface (Go/Java) or an abstract base class (C++). A type opts in with impl Trait for Type; code can then accept "any type that implements Trait" without naming the concrete type. Traits power #[derive(...)] (Copy, Default, Debug, PartialEq are all traits), are required in scope to call their methods (e.g. Read for read_to_string), and underpin both static dispatch (generics with trait bounds, resolved at compile time) and dynamic dispatch (trait objects like Box<dyn Error>, resolved at runtime).
Type
A classification of values that, in Rust, encodes three things: the value's shape (layout and operations — primitives, tuples, structs, enums); its ownership behaviour (whether it moves or is Copy on assignment is a property of the type); and its capabilities (the set of traits it implements — e.g. a type is comparable with == only if it implements PartialEq). Rust is statically and strongly typed: every value has a type, usually inferred (annotate when ambiguous, e.g. let paths: Vec<String> = …), and checked before the program runs with no implicit coercions. The type system is where the safety guarantees live — absence is the type Option<T>, failure is Result<T, E>, and &T vs &mut T are distinct types with different rules. Orientation
Unit test (#[test] / #[cfg(test)])
A function marked #[test], usually inside a #[cfg(test)] mod tests module that is compiled only when running tests. cargo test runs every such function and reports pass/fail. Lesson 3
