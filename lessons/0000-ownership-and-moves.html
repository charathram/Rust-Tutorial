The Ownership Model

How Rust manages memory without a garbage collector, and the move semantics behind its most common beginner error. Read this before you write any Rust.

How to use this primer This is the only theory the course front-loads. Read it once for the mental model, then start building in Lesson 1. You do not need to master borrowing, lifetimes, or the borrow checker here; each is taught when a project first needs it (Modules 2 onward). This is orientation, not a lesson to complete.

What this primer covers

After reading, you should be able to:

  • State Rust's three rules of ownership.
  • Explain what a move is, and why assigning a heap-owning value invalidates the original binding.
  • Distinguish Copy (stack) types from move-only (heap-owning) types.
  • Read the borrow of moved value compiler error and resolve it with clone() or a borrow.

The problem ownership solves

Every language has to answer one question: who frees a value's memory, and when? The three common answers:

  • C: the programmer, by hand (free). Miss it and you leak; do it twice and you corrupt the heap.
  • Go: a garbage collector, at runtime, at a time it chooses.
  • Rust: the compiler, at compile time, using a rule called ownership. There is no GC and no manual free, and it costs nothing at runtime.

This is the feature that distinguishes Rust. The borrow checker you may have heard about is simply ownership being enforced.

The three rules of ownership

From the official Book (Ch. 4.1):

  1. Each value in Rust has an owner.
  2. There can only be one owner at a time.
  3. When the owner goes out of scope, the value is dropped (freed).

Rule 3 is the automatic free: at the closing }, the compiler inserts the cleanup (see Drop) deterministically, not "eventually".

fn main() {
    let s = String::from("hello"); // s owns heap-allocated text
    // ... use s ...
}                                  // scope ends → s is dropped, memory freed

Moves: why assignment can invalidate a value

A String is a three-field handle on the stack (pointer, length, capacity) referring to text on the heap. Consider:

let s1 = String::from("hello");
let s2 = s1;          // the handle is copied; the heap text is NOT
println!("{s1}");     // ❌ error[E0382]: borrow of moved value: `s1`

If both s1 and s2 remained valid, both would free the same heap buffer at scope end: a double free, the classic C defect. Rust's resolution follows from rule 2 (one owner): the assignment moves ownership to s2 and marks s1 invalid. Any later use of s1 is a compile error.

Comparison: the same line in C, Go, and Rust Given b = a; where a holds a string:
  • C: shallow pointer copy; both refer to one buffer; avoiding the double free is your responsibility.
  • Go: both reference the same backing array; the GC frees it once nothing points at it.
  • Rust: a is moved into b and becomes unusable; the double free is impossible by construction, verified before the program runs.

Copy types: the exception

Small, fixed-size, stack-only values are duplicated on assignment, and both bindings remain valid:

let x = 5;
let y = x;
println!("{x} {y}");  // ✓ fine — i32 is Copy, x is still valid

Types that implement Copy include integers, bool, char, floats, and tuples of such types. The rule of thumb:

  • Lives entirely on the stack → it copies.
  • Owns data on the heap (e.g. String) → it moves.
Two escape hatches
  • Need both bindings to own the data? Call a.clone() for an explicit deep copy.
  • Need to use a value without taking it? Borrow it with &a. You'll do this in Lesson 1, where reading a file uses &path.
Rust never clones implicitly: a deep copy happens only when you ask for one.
