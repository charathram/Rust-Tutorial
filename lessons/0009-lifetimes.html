Lifetimes: borrowed data in types

You wrote first_chars(s: &str) -> &str in Lesson 8 and Source::name(&self) -> &str back in Lesson 6 — both return a reference borrowed out of their input, and both compiled without a word about how long that borrow lasts. A lifetime is the compiler's name for exactly that: how long a reference stays valid. This lesson makes it visible. You will read the annotation 'a, learn the elision rules that let you usually omit it, and hit the one place it becomes mandatory — a struct that holds a &str. You build that struct: a Report<'a> that borrows its label instead of owning it.

Lesson goals

By the end of this lesson you will be able to:

  • Say what a lifetime is — a region of code for which a reference is valid — and read the 'a annotation.
  • State the three elision rules and explain why name and first_chars need no annotation.
  • Write a struct that stores a reference, giving it a lifetime parameter (Report<'a>), and know why the compiler demands one.
  • Recognise E0106 (missing lifetime) and E0515 (returning a borrow of a local) and fix them.
  • Explain 'static and why a string literal has it.

Why lifetimes, why now

Every reference in Rust is valid only for a stretch of the program — from where it is created to where the thing it points at goes away. The borrow checker has enforced that since Lesson 4: a reference may never outlive its owner. A lifetime is simply the compiler's name for that stretch. Most of the time you never write one, because the compiler infers it. You have been leaning on that inference for eight lessons.

It stops being invisible the moment a type stores a reference. rwc is about to do that. Right now main prints each source's counts and label inline and throws the pieces away. This lesson bundles them into a small type — a Report that pairs a source's Counts with its label — and the label is borrowed, not copied. A struct that holds a &str is exactly the case the compiler cannot figure out on its own, so it makes you name the lifetime. That is the whole reason this lesson exists here: Report is the first type in the course that has to carry a 'a.

Step 1 — What a lifetime is

A lifetime is a region of code. A reference is tagged with the region during which the data it points to is guaranteed to exist, and the borrow checker refuses any use of the reference outside that region. You already saw the failure mode in Lesson 5 — use a value after its owner drops and the borrow checker stops you. Lifetimes are the bookkeeping behind that check.

The annotation is an apostrophe and a name, conventionally 'a (read "tick-a"). It attaches to a reference type:

&str          // a string slice (lifetime elided — the compiler picks it)
&'a str       // a string slice valid for the lifetime named 'a
&'a mut T     // a mutable reference valid for 'a

'a is not a duration in seconds and not a value you compute. It is a label the compiler uses to relate references to one another — "this output borrows from that input," "this field borrows from whatever built the struct." You are not telling the compiler how long anything lives; you are telling it which things must live at least as long as which others. It checks the rest.

Step 2 — Elision: why you rarely write one

If every reference has a lifetime, why have you never typed one? Because of lifetime elision: three rules the compiler applies to fill in the obvious cases so you don't have to. The rules, in order:

  1. Each elided lifetime in the parameters gets its own distinct lifetime.
  2. If there is exactly one input lifetime, it is assigned to every elided output lifetime.
  3. If one of the parameters is &self or &mut self, its lifetime is assigned to every elided output lifetime.

Two functions you already wrote are covered by these. Take Source::name. It has one reference parameter, &self, so rule 1 gives it a lifetime and rule 3 hands that same lifetime to the returned &str. Written out in full, the elided signature is:

pub fn name<'a>(&'a self) -> &'a str   // what the compiler infers from `fn name(&self) -> &str`

The explicit form says: the borrowed name lives as long as the borrow of self — which is right, since the name points into self.

Your turn — do the same for Lesson 8's first_chars. Its signature is fn first_chars(s: &str, n: usize) -> &str. It has no self; it has exactly one reference parameter (s — n is not a reference). Which rule assigns the output lifetime, and what is the explicit signature?

Show solution 
pub fn first_chars<'a>(s: &'a str, n: usize) -> &'a str

Rule 1 gives s the lifetime 'a; with exactly one input lifetime, rule 2 assigns it to the output. The returned slice borrows from s and cannot outlive it — which is exactly what first_chars does. The explicit and elided forms compile identically.

Elision only reaches the common shapes. When there is more than one input reference and no self, rule 2 doesn't apply and rule 3 doesn't apply, so the compiler cannot guess which input the output borrows from. Watch it refuse a function that takes two slices and returns one:

fn longest(a: &str, b: &str) -> &str {
    if a.len() >= b.len() { a } else { b }
}
error[E0106]: missing lifetime specifier
 --> src/lib.rs:1:33
  |
1 | fn longest(a: &str, b: &str) -> &str {
  |               ----     ----     ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value, but the
          signature does not say whether it is borrowed from `a` or `b`

The fix is to name the relationship yourself: tie both inputs and the output to one lifetime, saying "the result borrows from both, so it lives no longer than the shorter of the two."

fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}

rwc has no such function — but it is about to grow the other case elision never covers: a struct that stores a reference.

Step 3 — A struct that borrows: Report<'a>

The plan: a Report pairs one source's Counts with its label — the string main prints at the end of each row ("demo.txt", "file.md (markdown)", "total"). The Counts is Copy, so the Report owns it outright. The label is text that already lives elsewhere, so the Report borrows it — a &str field.

Try to write that the obvious way and the compiler stops you:

pub struct Report {
    label: &str,
    counts: Counts,
}
error[E0106]: missing lifetime specifier
 --> src/lib.rs:2:12
  |
2 |     label: &str,
  |            ^ expected named lifetime parameter
  |
help: consider introducing a named lifetime parameter
  |
1 ~ pub struct Report<'a> {
2 ~     label: &'a str,
  |

A struct that stores a reference must declare a lifetime parameter, for a concrete reason: the compiler has to guarantee the Report never outlives the text its label points into. The 'a is the handle it uses to enforce that — it means "this Report is valid only for the region 'a, the region its label is borrowed for." Elision can't help here: there is no function signature to read the borrow from, just a field. So you write it. The compiler even suggests the exact fix. Add the struct to lib.rs, near Counts:

// src/lib.rs — first_chars, Counts, and Source stay exactly as they are
/// One source's output row: its `Counts` plus a borrowed label.
///
/// The label is *borrowed* (`&'a str`), not owned, so a `Report` may not
/// outlive whatever its label points into. The `'a` lifetime names that
/// constraint — it ties the `Report` to the borrowed text.
///
/// ```
/// use rwc::{Counts, Report};
/// let counts = Counts::from_text("hello world\n");
/// let report = Report::new("hello.txt", counts);
/// assert_eq!(report.label(), "hello.txt");
/// assert_eq!(report.counts().words, 2);
/// ```
pub struct Report<'a> {
    label: &'a str,
    counts: Counts,
}

Now the impl block. The impl also carries the lifetime — you write impl<'a> Report<'a>, declaring 'a after impl and using it on the type, the same way a generic parameter is declared then used. The constructor borrows the label and stores it:

impl<'a> Report<'a> {
    /// Builds a report borrowing `label` and holding a copy of `counts`.
    pub fn new(label: &'a str, counts: Counts) -> Report<'a> {
        Report { label, counts }
    }

    /// The borrowed label. Elision ties the returned `&str` to `&self`, so it
    /// borrows for as long as the caller holds the `Report`.
    pub fn label(&self) -> &str {
        self.label
    }
    // ... counts and line go here (below) ...
}

Notice label takes no annotation: it has &self, so elision rule 3 gives the returned &str the lifetime of &self. Only the struct and new — where the borrow first enters the type — need 'a spelled out.

Step 4 — Use Report in main

Now route the output through Report. In the loop, build a report from each source's label and counts, then print its line. The label comes from source.describe(), which returns a String; the Report borrows it, so the String has to live at least as long as the report — binding it to label first gives it a name that outlives the borrow. First the import:

use rwc::{Counts, Report, Source};
use std::env;
use std::error::Error;

Then the loop body. The read and from_text lines are unchanged; the inline println! becomes three lines:

Two things are worth naming. First, counts is still usable on the next line (total.add(counts)) even though you passed it to Report::new — because Counts is Copy, that was a copy, not a move. Second, Report::new(&label, counts) passes &label (a &String) where a &str is expected; the same deref coercion you met in Lesson 8 applies.

The total row uses Report too. Its label is the literal "total":

    if sources.len() > 1 {
        println!("{}", Report::new("total", total).line());
    }

That literal is where 'static shows up — the subject of the next step.

Step 5 — Dangling borrows, and 'static

The 'a on Report is not decoration; it is a rule the compiler enforces. To see it bite, try to write a function that builds a Report from a String it owns and returns it:

fn report_for(name: &str) -> Report<'static> {
    let owned = String::from(name);
    Report::new(&owned, Counts::default())
}
error[E0515]: cannot return value referencing local variable `owned`
 --> src/lib.rs:7:5
  |
7 |     Report::new(&owned, Counts::default())
  |     ^^^^^^^^^^^^^------^^^^^^^^^^^^^^^^^^^^^
  |     |           |
  |     |           `owned` is borrowed here
  |     returns a value referencing data owned by the current function

The Report would borrow owned, but owned is dropped when the function returns — so the returned Report would point at freed memory. In a language without lifetimes this compiles and corrupts at runtime; here it is a compile error. The 'a is what lets the compiler catch it: it can see the Report's lifetime would have to outlive owned, which is impossible. (The fix is to make the caller own the text and lend it in, exactly as main does with label.)

So why does Report::new("total", total) in main work? Because a string literal is special: it is baked into the compiled binary and lives for the entire run of the program. Its type is &'static str — the lifetime 'static means "valid for the whole program." A &'static str satisfies any lifetime a Report could need, so Report::new("total", total) is a Report<'static> and never dangles.

'static, briefly 'static is the longest possible lifetime: data that lives from the point it exists until the program ends. String literals are the everyday example. It is a real lifetime like any other, just the maximal one — not a keyword you sprinkle on to silence errors. (You will meet the trait bound T: 'static later, under threads and async; it is related but distinct.)

Outcome rwc now has its first type that borrows: a Report<'a> holding a borrowed label beside an owned Counts. You can read the 'a annotation, you know the three elision rules that hide it in name and first_chars, and you know the one place it becomes mandatory — a struct that stores a reference. Optional extensions

  1. Two-lifetime struct. Give Report a second borrowed field with its own lifetime — say a borrowed note: &'b str — so the struct becomes Report<'a, 'b>. Notice you only need separate lifetimes if the two fields can be borrowed from different places for different durations; if they always come together, one lifetime is simpler.

Check your understanding

Why does struct Report { label: &str } fail to compile?

Right — the compiler needs a lifetime to guarantee the Report never outlives the text its label borrows, and elision only applies to function signatures, not struct definitions. The fix is Report<'a> with label: &'a str.

You can store references in structs — you just have to name the lifetime, because elision doesn't apply to struct definitions. The compiler needs 'a to stop the Report from outliving its borrowed label.

In fn longest<'a>(a: &'a str, b: &'a str) -> &'a str, what does 'a tell the compiler?

Right — 'a ties both inputs and the output together: the result borrows from the inputs, so the caller must keep both alive for as long as it holds the returned slice. Elision can't infer this because it can't tell which input the output comes from.

A lifetime says nothing about length or copying — 'a relates the references: the output borrows from a or b and therefore can't outlive either. That relationship is what elision couldn't guess with two input references.

Report::new("total", total) — the label is the literal "total". What is its lifetime?

Right — a string literal is &'static str: baked into the binary, valid for the entire run. That satisfies any lifetime a Report could require, so this never dangles.

String literals aren't temporaries — they live in the compiled binary for the whole program, so their type is &'static str. Every reference has a lifetime, literal or not; this one is the maximal 'static.
