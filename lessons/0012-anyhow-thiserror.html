Ergonomic errors: anyhow & thiserror

Back in Lesson 2 you returned Box<dyn Error> from main and, in Lesson 11, added a file's name to a read failure by hand: .map_err(|e| format!("could not read {}: {}", …)). That works, but it flattens the error: it glues the context and the cause into one string and throws the original error away. This lesson replaces that with the two crates most real Rust programs reach for: thiserror to give the library a proper typed error, and anyhow to carry context-rich errors through the binary. The anchoring feature: each failure says which file and why, as a real error chain, and you never have to format the strings yourself.

Lesson goals

By the end of this lesson you will be able to:

  • Explain the division of labour: thiserror for a library's own typed, matchable error; anyhow for an application's dynamic, context-carrying error.
  • Define an error enum with #[derive(thiserror::Error)], an #[error("…")] message per variant, and #[from] to convert an underlying error automatically.
  • Return anyhow::Result<()> from main and attach context with .context(…) / .with_context(|| …).
  • Read the resulting error chain (a top-level message plus its Caused by: source) and say why it beats a flattened string.
  • Contrast all of this with the Box<dyn Error> approach from Lesson 2: what each buys and costs.

Two crates, two jobs

Rust's standard library gives you the Error trait and ?, but nothing for the two things applications actually need: a concise way to define error types, and a way to add context as an error travels up. Two crates by the same author fill those gaps, and the split matches the shape rwc already has: a library (lib.rs) and a binary (main.rs):

  • thiserror goes in the library. It derives the Error trait for an enum you define, so rwc's own failures (a bad argument, a failed read) become a real type, RwcError, that a caller can match on. You write the variants; the macro writes the boilerplate.
  • anyhow goes in the binary. Its anyhow::Error is a smarter Box<dyn Error>: it holds any error, but also lets you staple a human message onto it with .context() while keeping the original as the cause. main doesn't care what concrete error came back; it just wants to add "which file" and report cleanly.

The rule of thumb the ecosystem settled on: libraries return thiserror types (callers may need to tell failures apart), applications use anyhow (they mostly add context and print). rwc is both, so it uses both.

Spaced review — vs. Box<dyn Error> (Lesson 2) Box<dyn Error> was the catch-all that let ? unify many error kinds into one return type. anyhow::Error does the same job and adds two things it lacked: a .context() chain, and a captured backtrace. And where Box<dyn Error> is opaque (you can't tell what went wrong without downcasting), a thiserror enum is transparent to callers: they can match on the variant. Rough map: Box<dyn Error> ≈ a bare anyhow::Error with no context; a hand-written enum ≈ what thiserror generates for you.

Step 2 — A typed library error with thiserror

Right now the library is dishonest about its errors. Source::from_args returns Result<…, Box<dyn Error>> and builds the error with format!("invalid argument: {flag}").into(), a stringly-typed error a caller can read but never inspect. And Source::read returns io::Result. Give the library one honest error type instead. Open lib.rs and, at the top, bring in the derive macro:

use std::fs;
use std::io::{self, Read};
use thiserror::Error;
use unicode_segmentation::UnicodeSegmentation;

Now define the error enum. Each variant is one way the library can fail; the #[error("…")] attribute is that variant's Display text (the message a user sees), and the macro writes the Error and Display impls for you. Two attributes are new and worth naming before you use them:

  • #[from] on a field asks the macro to generate a From impl, so the ? operator will convert that underlying error into this variant automatically. That is the same conversion Box<dyn Error> got for free, but now into your type.
  • #[error(transparent)] says "don't add a message of my own; forward Display and the source straight through to the wrapped error." Use it for a variant that only re-packages another error, like an I/O failure.

Here is the first variant, worked; the second is yours:

// src/lib.rs — add near the top, below the imports
/// Everything that can go wrong inside the `rwc` library.
#[derive(Debug, Error)]
pub enum RwcError {
    /// An argument began with `-` but was neither `-` nor a valid flag.
    #[error("invalid argument: {0}")]
    InvalidArgument(String),

The {0} in #[error("invalid argument: {0}")] interpolates the variant's first (and only) field, exactly like format!. So RwcError::InvalidArgument("-x".into()).to_string() is "invalid argument: -x", and you get Display without writing an impl.

Step 3 — Context-rich errors in main with anyhow

The library now hands back a precise RwcError. The binary's job is different: propagate whatever comes, add the one piece of context the user needs (which source), and report. That is exactly the job anyhow was built for. Change the imports at the top of main.rs: bring in anyhow's Result and its Context trait, and drop the now-unused std::error::Error:

use anyhow::{Context, Result};
use clap::Parser;
use rwc::{Columns, Counts, Report, Source};

Then the signature and the read. anyhow::Result<()> is just an alias for Result<(), anyhow::Error>. The ? on Source::from_args now works because RwcError implements Error (thiserror derived it), and anyhow::Error can absorb any such type. On the read, replace the hand-rolled .map_err(|e| format!(…)) with .with_context(…):

fn main() -> Result<()> {
    let cli = Cli::parse();
    let cols = Columns::from_flags(cli.lines, cli.words, cli.bytes, cli.chars, cli.graphemes);
    let sources = Source::from_args(cli.paths)?;

Show solution 
            .with_context(|| format!("could not read {}", source.name()))?;

Two forms exist: .context("message") takes a value, and .with_context(|| …) takes a closure that runs only if there is an error. Use the closure form when building the message costs something, like the format! here. Either way, anyhow keeps the original error as the source and stacks your message on top. That is the whole difference from Lesson 11's map_err: format!("… {}: {}", name, e) destroyed the io::Error to make a string; with_context preserves it.

Step 4 — Read the error chain

Tests first. The one new unit test brings the suite to 14; the doc-tests stay at 11:

   Doc-tests rwc
running 11 tests

A normal run is unchanged; errors only show up when something fails:

printf 'a b c\n' > demo.txt
cargo run -q -- demo.txt
     1      3      6      6      6 demo.txt

Now ask for a file that isn't there. Because main returns an anyhow::Result, Rust prints the returned error with its full chain (your context on top, the underlying cause beneath) on stderr, and exits non-zero:

cargo run -q -- demo.txt missing.txt
     1      3      6      6      6 demo.txt
Error: could not read missing.txt

Caused by:
    No such file or directory (os error 2)

Compare that with what Lesson 11 printed for the same failure, the flattened, quote-wrapped string a Box<dyn Error> built from format! produces:

# Lesson 11 (before):
Error: "could not read missing.txt: No such file or directory (os error 2)"

# Lesson 12 (after):
Error: could not read missing.txt

Caused by:
    No such file or directory (os error 2)

Same information, but the chain is structured: the context and the cause are separate lines, and the real io::Error is still in there (a program could inspect it), not melted into text. The invalid-argument path shows your thiserror message directly. clap parses flags first, so you reach it by forcing an argument through with --:

cargo run -q -- -- -x
Error: invalid argument: -x
Outcome rwc now handles errors the way real Rust programs do. The library returns a typed RwcError built with thiserror (an #[error] message per variant, #[from] for automatic io::Error conversion), so callers get a concrete, matchable type instead of a stringly-typed box. The binary returns anyhow::Result and stacks per-file context with .with_context(), producing a real error chain (which file, then why) instead of a flattened string, while preserving the underlying cause. You also saw that focused crates like these cost almost nothing: three packages, versus clap's twenty.

Optional extensions

  1. See the whole chain inline. anyhow's Display shows only the top message; its Debug (what main prints) shows the chain. Try eprintln!("{e:#}") on an error somewhere; the # "alternate" flag prints the chain on one line, context: cause. Compare the three forms.
  2. Add a variant. Give RwcError an Empty variant (#[error("no input given")]) and return it from a new check. Because it is a real enum, a caller could match on RwcError::Empty, which Box<dyn Error> could never offer.
  3. Turn on backtraces. Run RUST_BACKTRACE=1 cargo run -q -- missing.txt. anyhow captures a backtrace at the point the error is created; the env var makes it print. Note when it helps (a deep call stack) and when it is noise (a one-line failure like this).
  4. Where would anyhow be wrong? If rwc were a library other crates depended on, returning anyhow::Error from its public functions would force their errors to be opaque too. Explain in a sentence why public library APIs should return thiserror types, not anyhow.

Check your understanding

Why does rwc use thiserror in lib.rs but anyhow in main.rs?

Right. A library should hand back a concrete, inspectable type (its callers may need to tell failures apart), which is what thiserror generates. An application mostly propagates and reports, so anyhow's dynamic, context-carrying error is the ergonomic choice.

The split is about who consumes the error. Libraries return typed thiserror enums so callers can match; binaries use anyhow to add context and print. Neither restriction is compiler-enforced; it's a deliberate design rule.

What does .with_context(|| format!("could not read {}", source.name())) do that Lesson 11's .map_err(|e| format!("… {}: {}", name, e)) did not?

Right. map_err with format! flattened the context and the cause into one string, discarding the io::Error. with_context stacks the message on top and preserves the source, so you get a structured chain and can still reach the real error.

The point isn't speed or brevity. The old map_err(format!) destroyed the underlying io::Error to make a flat string; with_context keeps it as the cause, giving a real error chain.

In Io(#[from] io::Error), what does the #[from] attribute generate?

Right. #[from] asks thiserror to write impl From<io::Error> for RwcError. That's what lets fs::read_to_string(path)? turn an I/O failure into RwcError::Io with no manual mapping.

It generates From<io::Error> for RwcError, which is the direction the ? operator needs to lift an io::Error into your error type. It has nothing to do with Default or converting back to io::Error.

Primary sources

The two crates' docs are short and worth reading in full: the anyhow documentation (see the Context trait and the "details" section on the error chain) and the thiserror documentation (every attribute: #[error], #[from], #[source], transparent). For the underlying model both build on, read The Rust Book, Chapter 9 — "Error Handling." The design rule of thumb (thiserror for libraries, anyhow for applications) is stated directly in both crates' READMEs by their author.
