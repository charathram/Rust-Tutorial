Error handling: from panics to Result

Replace the .expect() panics in rwc with real error handling: clean messages on standard error and correct exit codes, the way a command-line tool should behave.

Lesson goals

By the end of this lesson you will be able to:

  • Explain why .expect() is unsuitable for a finished tool.
  • Return a Result from main and propagate errors with the ? operator.
  • Convert an Option (a missing argument) into a recoverable error.
  • Produce clean error output on stderr with a non-zero exit code.
Starting point Work in 0002-rwc/, which already contains the finished tool from Lesson 1. As before, the lines highlighted in green are the ones to change.

Why .expect() is not enough

The current tool calls .expect(), which panics on failure. Run it against a file that does not exist:

cargo run -- nope.txt; echo "exit=$?"
thread 'main' panicked at src/main.rs:7:42:
could not read file: Os { code: 2, kind: NotFound, message: "No such file or directory" }
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
exit=101

Three problems for a real tool:

  • A panic is for unrecoverable bugs, not expected conditions like a missing file.
  • The output mentions internal file/line numbers and a backtrace hint, which is noise for a user.
  • The exit code is 101 (Rust's panic code), not the conventional 1 that scripts and shells expect on failure.

Step 1 — Return a Result from main and propagate with ?

Apply these five changes to src/main.rs:

use std::env;
use std::fs;
use std::error::Error;

fn main() -> Result<(), Box<dyn Error>> {
    let path = env::args().nth(1).ok_or("usage: rwc <file>")?;
    let text = fs::read_to_string(&path)?;

    let lines = text.lines().count();
    let words = text.split_whitespace().count();
    let chars = text.chars().count();

    println!("{lines:>6} {words:>6} {chars:>6} {path}");
    Ok(())
}

What each change does, and why it is needed:

  • use std::error::Error; The standard library defines a trait named Error that every error type implements. This line brings that name into scope so we can refer to it in main's new return type on the next line; without it we would have to spell out the full path std::error::Error wherever the type appears.
  • fn main() -> Result<(), Box<dyn Error>>: until now main returned nothing. We change its signature so it returns a Result, which is how any Rust function announces "this might fail." The success variant carries () (the unit type, meaning "no meaningful value," Rust's equivalent of a void return) because when main succeeds it has nothing useful to hand back. The failure variant carries Box<dyn Error>, the catch-all error type explained in the callout below. This new return type is also what makes the ? operator legal inside main.
  • .ok_or("usage: rwc <file>")?: nth(1) hands back an Option, which is None when no argument was supplied. The ? operator works on a Result, not an Option, so we convert first: .ok_or(msg) turns Some(path) into Ok(path) and turns None into Err(msg), using our usage text as the error. The trailing ? then either unwraps the path (on Ok) or stops main and returns the error (on Err). Together they replace the old .expect() with a clean usage error instead of a panic.
  • fs::read_to_string(&path)? is the main change. read_to_string already returns a Result. Appending ? means: if the read succeeded, give back the String and continue; if it failed (missing file, no permission), immediately return that error out of main. On success it behaves exactly like the old .expect(); on failure it propagates a recoverable error rather than crashing.
  • Ok(()): because main now promises a Result, every path through it must produce one. Reaching the final line without having returned an error means success, so we hand back Ok(()), the success variant wrapping the unit value. Leave this off and the program will not compile, because main would be declaring a Result it never actually returns.
Concept: the ? operator Applied to a Result, ? means "if this is Ok(v), give me v; if it is Err(e), stop and return Err(e) from the enclosing function." It replaces a verbose match, and it requires the enclosing function to return a compatible Result, which is why main's signature had to change. (Rust Book, Ch. 9.2.)
Concept: Box<dyn Error>, a trait object Different operations fail with different error types (an IO error here, a custom message there). Box<dyn Error> is one return type that accepts any value implementing the Error trait; the concrete type is resolved at runtime. The ? operator converts each specific error into it automatically, so main stays simple while individual steps fail in their own ways.

Step 2 — Verify the behaviour

Run the three cases and inspect the exit code each time with echo $?:

cargo run -- ../MISSION.md; echo "exit=$?"   # counts, then exit=0
cargo run -- nope.txt;      echo "exit=$?"   # error to stderr, exit=1
cargo run;                  echo "exit=$?"   # usage error, exit=1

The missing-file run now prints a single clean line to standard error and exits 1:

Error: Os { code: 2, kind: NotFound, message: "No such file or directory" }
exit=1
On the message format When main returns an Err, Rust prints it to stderr using its debug representation (hence the Os { … } struct and the quotes around the usage message) and exits with code 1. This is correct and scriptable, if a little raw. The first two extensions below show how to produce polished, user-facing messages.

Optional extensions

  1. Add context to the file error. Include the path in the message using map_err: 
    let text = fs::read_to_string(&path).map_err(|e| format!("{path}: {e}"))?;
    A String converts into Box<dyn Error>, so this compiles and yields a clearer message.
  2. Adopt the anyhow crate. The official CLI guide recommends anyhow for ergonomic error context and cause chains (main() -> anyhow::Result<()> with .with_context(…)). We will add dependencies in Lesson 3. (CLI Book — Error handling.)
  3. Use exit code 2 for usage errors. Many tools reserve 2 for incorrect invocation. Match on the argument explicitly and call std::process::exit(2) after printing usage to stderr with eprintln!.

Check your understanding

Applied to an Err, what does the ? operator do?

On Err, ? returns early and hands the error back to the caller: here, out of main, where Rust prints it and exits 1.

No: ? returns the Err from the enclosing function (it does not panic or ignore). The caller decides what happens next.

Why did main's signature have to change to use ??

? returns an Err from the enclosing function, so that function must return a Result. That is why main now returns Result<(), Box<dyn Error>>.

It is about propagation: ? returns an Err from the function, so the function must return a Result. Speed and mutability are unrelated.
