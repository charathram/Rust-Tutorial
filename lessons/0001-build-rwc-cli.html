Build a command-line tool: rwc

A working reimplementation of Unix wc, which counts the lines, words, and characters in a file, built incrementally in four steps.

Lesson goals

By the end of this lesson you will be able to:

  • Create and run a Rust binary with cargo.
  • Read command-line arguments via std::env::args.
  • Read a file into a String with std::fs::read_to_string.
  • Handle absent or failing values using Option, Result, and .expect().
  • Count lines, words, and characters with the iterators a string provides.

The result is a working rwc tool you can run on any text file.

How to work through this
  • The project is scaffolded at 0001-rwc/ (named to match this lesson). Open a terminal there; the only file you edit is src/main.rs.
  • Run with cargo run -- <args>. The -- separates cargo's arguments from your program's: everything after it is passed to rwc.
  • Run cargo run after every step, so you only ever debug a few new lines at a time.
  • From Step 2 on, lines highlighted in green are those added or changed since the previous step. Type or edit only those.

Step 0 — Run the generated project

cargo new left a hello-world in src/main.rs. Confirm the toolchain works before changing anything:

cargo run

You should see Hello, world!. Cargo compiled src/main.rs to target/debug/rwc and ran it. Compile and run is the whole workflow.

Step 1 — Read a command-line argument

Replace the contents of src/main.rs with:

use std::env;

fn main() {
    let path = env::args().nth(1).expect("usage: rwc <file>");
    println!("counting: {path}");
}

What each part of the fourth line does, left to right:

  • env::args() returns the program's command-line arguments, but not as an array. It gives you an iterator: a lazy sequence you pull values from one at a time, rather than a collection holding them all at once. (If you know Python generators, it is the same idea.) The methods that follow are how you draw a value out of that iterator.
  • .nth(1) pulls the item at position 1 from the iterator. Positions are zero-based, and position 0 is always the path to the program itself, so the first argument a user actually typed is at position 1: the file path we want.
  • .expect("…") covers the case where there is no argument at position 1, which means the previous step cannot return a plain value (the next callout explains exactly what it returns). .expect returns the value if one is present; if not, it stops the program immediately and prints this message. For now this is how we handle a missing argument.
  • println!("… {path}") prints a line of text. The ! tells you println is a macro, not a function. For now, just read ! as "this is a macro" and move on. The {path} inside the string is filled in with the value of the path variable, the same way an f-string works in Python or a template literal in JavaScript.

Run it both ways:

cargo run                  # no arg → crashes with our usage message
cargo run -- ../MISSION.md  # prints: counting: ../MISSION.md
Concept: Option and .expect() There may be no second argument, so .nth(1) cannot return a plain String. It returns an Option<String>: either Some(path) or None. Rust requires you to address the None case before using the value, which eliminates an entire class of null/nil defects. .expect("…") handles None by stopping the program; Lesson 2 replaces it with graceful handling.

Step 2 — Read the file's contents

Add the second use line and the two new body lines:

use std::env;
use std::fs;

fn main() {
    let path = env::args().nth(1).expect("usage: rwc <file>");
    let text = fs::read_to_string(&path).expect("could not read file");
    print!("{text}");
}

What each new line does:

  • fs::read_to_string(&path) opens the file at path and reads its entire contents into a single String in memory. For the files we are counting, reading everything at once keeps the program simple.
  • &path: the & passes the function a borrowed view of path instead of handing the value over. The function only needs to read the path, so a borrow is enough, and path stays usable afterward. The blue callout below connects this to the ownership rule from the primer.
  • .expect("could not read file"): reading a file can fail for ordinary reasons. The file may not exist, or you may lack permission. As with the argument in Step 1, the read does not return a plain String. It returns a value that is either the contents or an error, and .expect takes the contents or, on failure, stops the program with this message. (Lesson 2 replaces this with proper error handling.)
  • print! (no ln) prints its text with no trailing newline added. We use print! rather than println! here because the file's contents already end in a newline, and we do not want to add a second blank line.
cargo run -- ../MISSION.md   # dumps the file back at you
cargo run -- nope.txt        # crashes: could not read file
Connection: ownership in practice (Orientation primer) The & in &path is a borrow. read_to_string only needs to read the path, so it borrows it and returns it. Passing path by value would move it into the function, leaving path unusable on the next line. To confirm, delete the & and run: the resulting error is the ownership rule from the Orientation primer, resolved with a single character.

Step 3 — Count lines, words, and characters

A String provides iterators over lines, words, and characters; counting each is a call to .count(). Replace the print! line with:

use std::env;
use std::fs;

fn main() {
    let path = env::args().nth(1).expect("usage: rwc <file>");
    let text = fs::read_to_string(&path).expect("could not read file");

    let lines = text.lines().count();
    let words = text.split_whitespace().count();
    let chars = text.chars().count();

    println!("{lines:>6} {words:>6} {chars:>6} {path}");
}

What the new lines do (each builds on the iterator idea from Step 1):

  • text.lines() gives an iterator that yields the text one line at a time. On its own it counts nothing; chaining .count() walks the whole iterator and returns how many items it produced, here the number of lines.
  • text.split_whitespace() gives an iterator over the runs of non-whitespace between spaces, tabs, and newlines: in other words, the words. .count() then tallies them. This matches how wc defines a "word."
  • text.chars() gives an iterator over the text's characters (Unicode scalar values), and .count() tallies those. The callout below explains why this counts characters rather than bytes.
  • {lines:>6}: inside the format string, this is the same {variable} interpolation as before, plus a format specifier after the colon. >6 means "right-align this value in a field at least 6 characters wide," which lines the three numbers up into neat columns the way wc does.
cargo run -- ../MISSION.md

Compare against the reference implementation. Lines and words should match exactly:

wc ../MISSION.md   # prints: lines words bytes
Why .chars().count(), not .len() text.len() returns bytes, not characters. Rust strings are UTF-8, where one character may span several bytes: "வணக்கம்" from your hello2 project is far more bytes than characters. .chars() counts characters. Because wc counts bytes by default, its final column exceeds your chars column on any file containing non-ASCII text (em-dashes, accents, Tamil, Hindi). The two tools are measuring different things. Even MISSION.md shows a small gap, because its — dashes are 3 bytes each.
Outcome You have a working command-line tool that parses an argument, reads a file, and reports lines, words, and characters with correct Unicode handling. cargo run -- <file> now produces real output, and you can account for every line of it.

Optional extensions

  1. Byte count. Add let bytes = text.len(); and a fifth column so the output matches wc exactly on an ASCII file.
  2. Longest line. Report the character length of the longest line: 
    let longest = text.lines().map(|l| l.chars().count()).max();
    |l| l.chars().count() is a closure (an inline anonymous function; l is each line). .map(…) transforms each line into its length and .max() returns the largest as an Option (it is None for an empty file), so handle it with .expect() or a default.
  3. Clearer message. Revise the .expect text, or remove the argument entirely and compare the run-time crash with a compile-time error.

Check your understanding

Why does fs::read_to_string return a Result rather than a plain String?

IO can fail (missing file, permissions), so the type forces you to acknowledge the error path before using the value.

It's about fallibility: file IO can fail, so Result makes you handle the error case before reaching the String.

Why pass &path instead of path to read_to_string?

Passing by value would move path into the function; borrowing with & lets the function read it while you keep ownership.

It's ownership: passing path by value moves it away; & borrows so you can still use path afterward.

On a file of Tamil text, why might your chars count differ from wc's count?

UTF-8 uses multiple bytes per non-ASCII character, so byte count (wc default) exceeds the character count from .chars().

No bug: UTF-8 stores non-ASCII characters in several bytes, so wc's byte count is larger than .chars()'s character count.
