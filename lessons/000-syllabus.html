Rust · Course overview

Syllabus

A project-anchored path from the ownership model to shippable command-line tools and web backends — with explicit outcomes, prerequisites, checkpoints, and staged assessment.

This course builds real, runnable programs. Each module declares what you will be able to do, states its prerequisites, and ends in a checkpoint deliverable with acceptance criteria. Two cumulative checkpoints interleave everything learned so far. The glossary holds shared vocabulary; curated sources back every claim.

The mission

Get to where you reach for Rust by default — building CLI utilities and web backends that are fast, correct, and memory-safe without a garbage collector. The course is assessed against three success criteria: ship a non-trivial CLI end to end; stand up a small web service; read and resolve ownership, borrowing, and error-handling code without guessing.

How the course works

  • Project-anchored, with one exception. A single Module 0 primer front-loads the ownership model up front. From the first numbered lesson onward, every concept is introduced by a concrete feature of the tool currently being built — no theory taught in isolation.
  • Outcomes, prerequisites & checkpoints. Each module lists measurable outcomes, the modules it assumes, and a deliverable judged against stated acceptance criteria. Within a module, lessons are strictly sequential — so module prerequisites plus that ordering define the full dependency graph.
  • Spaced & genuinely interleaved. Modules carry a review step that recalls earlier material in new work, and two Interleaved checkpoint problem sets (after Modules 4 and 6) mix tasks drawn at random from all prior modules — the strict sense of interleaving, not just spaced repetition.
  • Practice in the wild. Each module names a community touchpoint (users.rust-lang.org, r/rust) where you can post your work and get real critique.
  • Reproducibility. Projects pin a toolchain (rust-toolchain.toml, currently 1.96) and lock crate versions. The web-stack modules (axum, sqlx) track fast-moving crates and will be version-refreshed when reached.

Scope & time commitment

36 lessons across 8 modules. Summing the effort tags gives roughly 50 hours of lesson work; with recommended primary-source reading and the staged capstone, plan on 70–90 hours total. These are estimates and shift with your pace and depth.

Module 0 · Orientation

The one concept to learn before writing any code — read once, then build. Not a project lesson.

After the primer you can:

  • State the three ownership rules and predict whether a snippet compiles.
  • Explain what a move is, and why assigning a heap-owning value invalidates the original.
  • Distinguish Copy (stack) types from move-only (heap-owning) types.
  • ○
    Orientation — The Ownership ModelMComplete
    A read-once primer. Borrowing, lifetimes, and the borrow checker are taught later, where a project needs them.

Prerequisites  None — start here.

Checkpoint  No deliverable — orientation only. The model is put to work for real starting in Module 1.

Module 2 · Structuring a program

Grow rwc into a multi-file, stdin-aware tool — one real feature per lesson, refactored while the tests stay green. Each new feature is what introduces the concept.

By the end you can:

  • Pass data by reference (&, &mut) and state the borrowing rules.
  • Diagnose and resolve common borrow-checker errors without guessing.
  • Model program data with structs and enums, and branch with exhaustive match.
  • Choose &str vs String deliberately, and read/write basic lifetime annotations.
  • 04
    Borrowing & referencesMComplete
    Anchor: rwc grows to count several files and total them — shared &, mutable &mut, and the shared-XOR-mutable rule, with the Lesson 3 tests as the safety net.
  • 05
    Reading borrow-checker errorsSComplete
    Anchor: deliberately provoke the four classic E03xx errors in rwc (E0596, E0382, E0502, E0499); read each message structurally and apply its standard fix.
  • 06
    Structs & enumsMComplete
    Anchor: a Counts struct replacing the (usize, usize, usize) tuple, and a Source enum for file vs stdin input. First lesson with src/lib.rs, so doc-tests land here.
  • 07
    Pattern matching with matchSComplete
    Anchor: the Source enum's match revisited in depth — exhaustiveness as a compile-time guarantee (add a variant, watch every match demand it), plus guards, bindings, and if let/let else.
  • 08
    Strings & slices: String vs &strMComplete
    Anchor: a character count added beside the byte count; String vs &str, slicing, and why a byte index can split a UTF-8 character.
  • 09
    Lifetimes: borrowed data in typesMComplete
    Anchor: a struct that holds a &str (a per-file Report<'a>) is exactly where 'a first becomes necessary; the elision rules and 'static. Advanced cases revisited under async (Module 6).

Prerequisites  Module 1.

Checkpoint  rwc counts multiple files (with a total line), reads stdin when given no path, and is built from named tested functions plus a Counts struct and Source enum. The Module 1 tests still pass; zero clippy warnings.

Spaced review  Re-explain why &path (Lesson 1) was a borrow — now using the borrowing rules — and predict the error if the & is removed.

Practice  Run clippy; reconcile its suggestions with the borrowing rules you just learned.

Interleaved checkpoint A  after Module 4

Module 6 · Shared ownership & concurrency

The hardest part of the course: shared and interior-mutable state, threads, and the async model. This is what the web-server modules build on.

By the end you can:

  • Use Rc/Arc, RefCell/Mutex, and explain when each is needed.
  • Spawn threads, share data with Arc<Mutex<…>> and channels, and reason about Send/Sync.
  • Explain the async model — futures, .await, executors — and write async functions on tokio.

Interleaved checkpoint B  after Module 6

A cumulative exercise on your own wordfreq, drawing across Modules 1–6:

① Parallelize the hot loop with Arc<Mutex> or channels (M6) and justify each ownership move (M1).  ② Convert one path to async and explain the lifetime that appears (M2/M6).  ③ Route all failures through anyhow with context (M3).  ④ Feed it JSON or CSV input via serde (M5).  ⑤ Add an integration test for the concurrent path (M1/M4).
