Pattern matching with match

Go deeper on the match you already used in Source. Teach rwc to read standard input from a lone -, lean on the compiler's exhaustiveness check, and pick up guards, bindings, and the if let / let else shorthands.

Lesson goals

By the end of this lesson you will be able to:

  • Match on any value — an enum, or a plain string — with literal patterns and the _ wildcard.
  • Bind data out of a variant (pull the path out of Source::File).
  • Add a condition to an arm with a guard.
  • Rely on exhaustiveness: add an enum variant and the compiler points at every match that must handle it.
  • Reach for if let and let else when you care about a single variant.

Why match, and why now

You already wrote two match expressions in Lesson 6: Source::name and Source::read both branch on which variant self is. That is the common case, but match does more. This lesson uses it to add one real feature — reading stdin from a lone -, the way wc does — and then covers the parts you reach for constantly: exhaustiveness, guards, bindings, and the if let / let else shorthands.

Coming from C or Go, match is like switch with two differences that matter: the compiler checks you have covered every case, and a pattern can pull data out of the value it matches.

Step 1 — Match on a value: read stdin from -

Real wc treats a lone - argument as "read standard input here", so cat a | wc - b mixes piped input with a file. Right now from_args turns every argument into a Source::File:

    pub fn from_args(args: Vec<String>) -> Vec<Source> {
        if args.is_empty() {
            vec![Source::Stdin]
        } else {
            args.into_iter().map(Source::File).collect()
        }
    }

So far you have matched on an enum (self). But match works on any value, including a string. Here is name from Lesson 6 as a reminder of the shape — one arm per case, each arm a pattern => value:

    pub fn name(&self) -> &str {
        match self {
            Source::File(path) => path,
            Source::Stdin => "-",
        }
    }

Rewrite from_args so each argument is matched: the string "-" becomes Source::Stdin, and anything else becomes a Source::File.

    pub fn from_args(args: Vec<String>) -> Vec<Source> {
        if args.is_empty() {
            return vec![Source::Stdin];
        }
        args.into_iter()

Show solution 
    pub fn from_args(args: Vec<String>) -> Vec<Source> {
        if args.is_empty() {
            return vec![Source::Stdin];
        }
        args.into_iter()
            .map(|arg| match arg.as_str() {
                "-" => Source::Stdin,
                _ => Source::File(arg),
            })
            .collect()
    }

What changed, and why:

  • match arg.as_str() — arg is a String; the literal pattern "-" is a &str, so as_str() gives the &str to match against. The same match machinery works on a string as on an enum.
  • "-" => Source::Stdin — a literal pattern. The arm runs only when the argument is exactly -.
  • _ => Source::File(arg) — the wildcard _ matches every other string. match is an expression, so each arm produces the Source that .map collects. A string is an open set — infinitely many values — so a catch-all is required here; you can't spell out every string. Contrast Step 2, where Source is a finite enum you own and you'll drop the catch-all on purpose so the compiler flags any variant you add later.
  • return vec![Source::Stdin]; — an early return for the no-arguments case, so the rest of the function isn't nested in an else.

Step 2 — Exhaustiveness: add a variant, watch every match fail

The reason match is safe is that the compiler checks you have handled every case. To see it, add a third variant to Source — a URL source, say — and build:

pub enum Source {
    File(String),
    #[default]
    Stdin,
    Url(String),
}
cargo build

112 |         match self {
    |               ^^^^ pattern `&Source::Url(_)` not covered
    |
note: `Source` defined here
   --> src/lib.rs:67:10
    |
 67 | pub enum Source {
    |          ^^^^^^
...
 71 |     Url(String),
    |     --- not covered
help: ensure that all possible cases are being handled by adding a match arm
    |
114 ~             Source::Stdin => "-",
115 ~             &Source::Url(_) => todo!(),
    |

error[E0004]: non-exhaustive patterns: `&Source::Url(_)` not covered
   --> src/lib.rs:129:15
    |
129 |         match self {
    |               ^^^^ pattern `&Source::Url(_)` not covered

error: could not compile `rwc` (lib) due to 2 previous errors

Two errors, one per match: both name and read stop compiling until they handle Url. The message names the uncovered variant and even drafts the arm to add. This is the guarantee — you cannot forget a case, and the compiler tells you exactly where.

This works because name and read list explicit arms (File, Stdin). Had they ended in a catch-all _ => "-", the new Url would have fallen through it silently and the compiler would have said nothing. For enums you own, prefer explicit arms over _, so the compiler walks you through every match when the enum grows.

We are not adding URLs, so remove the Url line and rebuild to get back to green. The point was the guarantee, not the feature.

Step 3 — Bindings and guards

An arm like Source::File(path) => … does two things at once: it matches the File variant and binds the String inside it to path. That binding is how you read a variant's data — it is why name can return the path. You can also bind a value while testing it, with name @ pattern:

let n = 5;
match n {
    digit @ 0..=9 => println!("one digit: {digit}"),
    other => println!("{other}"),
}

A guard adds a condition to an arm, written pattern if condition. The arm runs only when the pattern matches and the guard is true:

let size = 1500;
let bucket = match size {
    0 => "empty",
    n if n < 1024 => "small",
    _ => "large",
};

Show solution 
    pub fn describe(&self) -> String {
        match self {
            Source::File(path) if path.ends_with(".md") => format!("{path} (markdown)"),
            Source::File(path) => path.clone(),
            Source::Stdin => "-".to_string(),
        }
    }

Order matters: the guarded File arm comes first, so a .md path is caught before the plain File arm. A guard does not count toward exhaustiveness — the compiler still requires the second, unguarded File arm to cover the files that are not markdown.

Step 4 — if let and let else

When you care about a single variant, a full match is noisy. if let runs a block only when one pattern matches:

let source = Source::File("a.txt".to_string());
if let Source::File(path) = &source {
    println!("counting {path}");
}

let else is the mirror image: it binds when the pattern matches, and when it does not, the else block must leave the scope (return, break, continue, or panic!). It is how you pull a value out and bail early in one line, with the binding staying in scope afterward.

Show solution 
    pub fn path_or_dash(&self) -> String {
        let Source::File(path) = self else {
            return "-".to_string();
        };
        path.clone()
    }

Note where path lives: after a let … else the binding is available for the rest of the function, not just inside a block. That is the difference from if let, whose binding is scoped to its block.

path_or_dash deliberately mirrors name from Lesson 6, with one difference: name returns a &str borrowed from self, while path_or_dash returns an owned String — reach for it when the value must outlive the Source it came from.

Optional extensions

  1. Reject unknown flags with a guard. An argument like -l currently becomes a Source::File("-l") that fails at read time with a confusing message. Add a guard arm flag if flag.starts_with('-') before the wildcard and report a clear error (you will need from_args to return a Result, rippling into main).
  2. Make the Url variant real. Add a Source::Literal(String) for text given inline, handle it in name and read (read just returns the string), and let the compiler walk you through every match that needs the new arm.
  3. Use matches!. When you only need a yes/no, matches!(source, Source::Stdin) is shorter than a full match. Add an is_stdin(&self) -> bool method with it.

Check your understanding

A match on Source handles only File and Stdin. You add a Url variant. What happens?

Right — match must be exhaustive, so the compiler rejects the code (and names the missing variant) until every case is handled.

Exhaustiveness is checked at compile time: the code won't build until Url is handled. There's no runtime panic and nothing is silently skipped — unless you used a _ catch-all, which absorbs new variants without warning.

What does the guard do in Source::File(path) if path.ends_with(".md") => …?

Right — a guard is an extra boolean condition. The pattern must match and the condition must be true. A later unguarded File arm is still needed for the non-markdown files.

A guard only tests a condition; it changes nothing. The arm fires when the pattern matches and the guard is true, so you still need a plain File arm for paths that don't end in .md.

In let Source::File(path) = source else { … };, what must the else block do?

Right — if the pattern doesn't match there is no path to bind, so the else must diverge. After a successful let else, the binding stays in scope.

There's no value to bind when the pattern fails, so the else can't fall through — it must diverge (return/break/continue/panic!).
