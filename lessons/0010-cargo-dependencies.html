Cargo, crates & dependencies

Every line of rwc so far has used only the standard library. That ends here. This lesson adds rwc's first external dependency, a crate from crates.io, and puts it to work counting a fifth column: graphemes, the user-perceived characters. The choice is deliberate. The standard library has no grapheme segmentation, on purpose: doing it correctly means shipping large Unicode tables the std team leaves to a dedicated crate. That is exactly what dependencies are for. Along the way you learn the machinery every real Rust project leans on: cargo add, the [dependencies] table, semver version requirements, the Cargo.lock lockfile, and pinning the compiler with rust-toolchain.toml so a build reproduces on any machine.

Why a dependency, why now

In Lesson 8 you added a chars count with one line of std: text.chars().count(). That counts Unicode scalar values: chars. But a char is not always a whole user-visible character. Write "é" as the base letter e followed by a combining acute accent (U+0301) and that is two chars but one thing a reader sees: one grapheme cluster. Emoji made of joined parts (a family, a flag, a skin-tone modifier) are many chars and one grapheme too.

Counting graphemes correctly is not a one-liner. It needs the Unicode text-segmentation algorithm and the data tables behind it: thousands of entries that change with each Unicode release. The standard library deliberately does not ship this; str has no .graphemes() method. The Rust project maintains it as a separate crate, unicode-segmentation, so you pull in exactly that capability without the std team owning it forever. Reaching for a crate here is the right call, not a shortcut. It is the perfect first dependency because the "why" is concrete: std can't do this, and a well-maintained crate can.

Step 1 — Add the crate with cargo add

A crate is a Rust package: a unit of code published to the registry crates.io and reusable by name. A dependency is a crate your project builds on. You declare dependencies in Cargo.toml, under the [dependencies] table. The easy way to add one is cargo add, which looks up the latest version and writes the entry for you:

cargo add unicode-segmentation
    Updating crates.io index
      Adding unicode-segmentation v1.13.3 to dependencies
             Features:
             - no_std
    Updating crates.io index
     Locking 1 package to latest Rust 1.96.0 compatible version
      Adding unicode-segmentation v1.13.3

Two things happened. First, Cargo.toml grew a line under [dependencies]:

[package]
name = "rwc"
version = "0.1.0"
edition = "2024"

[dependencies]
unicode-segmentation = "1.13.3"

(The Features note lists optional parts of the crate you can switch on. unicode-segmentation offers a no_std feature for environments without the standard library. You need none of them here; the defaults are what you want.)

Second, Cargo resolved and locked the version: "Locking 1 package." That is the Cargo.lock file, the subject of Step 3. First, what does "1.13.3" actually mean?

Step 2 — Semver: what a version requirement allows

Crates use semantic versioning (semver): a version is MAJOR.MINOR.PATCH, and each part signals what changed.

  • MAJOR (the 1): a breaking change; code written for the old version may stop compiling.
  • MINOR (the 13): new features, backward-compatible; old code keeps working.
  • PATCH (the 3): backward-compatible bug fixes only.

The bare string "1.13.3" in Cargo.toml is not "exactly 1.13.3." It is shorthand for a caret requirement, ^1.13.3, which means "at least 1.13.3, and anything below the next major version":

unicode-segmentation = "1.13.3"   // means ^1.13.3  →  >=1.13.3, <2.0.0

So Cargo may resolve to 1.13.4 or 1.20.0 when they exist (those are compatible), but never 2.0.0, which by semver is allowed to break you. The rule for caret: it permits changes that do not modify the left-most non-zero component.

That last phrase matters for pre-1.0 crates, which are common. When the major version is 0, the left-most non-zero component is the minor, so the minor becomes the breaking one:

foo = "0.5.2"   // ^0.5.2  →  >=0.5.2, <0.6.0   (minor is treated as breaking)

bar = "0.2.4"   // ^0.2.4  →  >=0.2.4, <0.3.0

The major is 0, so the left-most non-zero component is the minor (2). Caret locks it: Cargo may pick 0.2.5 or 0.2.99, but 0.3.0 is treated as a breaking release and is excluded. (For 0.0.z versions the patch is the only non-zero part, so ^0.0.3 allows only 0.0.3.)

Step 3 — The lockfile: Cargo.lock

Cargo.toml says what you allow (a range); Cargo.lock records what Cargo actually chose: one exact version per crate in the whole dependency tree, with a checksum. Open it after the cargo add and you will see two things changed: your rwc package now lists its dependency, and a new entry pins the exact crate Cargo downloaded:

# This file is automatically @generated by Cargo.
# It is not intended for manual editing.
version = 4

[[package]]
name = "rwc"
version = "0.1.0"
dependencies = [
 "unicode-segmentation",
]

[[package]]
name = "unicode-segmentation"
version = "1.13.3"
source = "registry+https://github.com/rust-lang/crates.io-index"
checksum = "c6f5d3c3b1bf09027a88a6bc961fc00497d651009560b5463668dc81b0fa87a8"

You never edit this file by hand; Cargo writes it. Its job is reproducible builds. Once the lock exists, cargo build uses the exact versions it records and does not re-resolve, even if 1.13.4 is published tomorrow. Commit Cargo.lock to version control so your machine, a teammate's, and CI all compile rwc against byte-for-byte the same dependencies. (The old advice was "commit the lock for binaries, not libraries"; current guidance is to commit it either way. rwc is a binary, so this was never in doubt.)

You can inspect the resolved tree at any time:

cargo tree
rwc v0.1.0 (/…/0010-rwc)
└── unicode-segmentation v1.13.3

To move within the semver range on purpose (pick up newer compatible versions and rewrite the lock), you run cargo update (or cargo update -p unicode-segmentation for just one crate). Nothing to update yet, but the dry run shows what it would do without touching the file:

cargo update --dry-run
    Updating crates.io index
     Locking 0 packages to latest Rust 1.96.0 compatible versions
warning: not updating lockfile due to dry run

The split is the whole point: Cargo.toml is the policy (which versions are acceptable), Cargo.lock is the fact (which versions you built with), and cargo update is the only routine command that changes the fact.

Step 4 — Use the crate: a grapheme count

Now spend the dependency. The crate adds its .graphemes() method to &str through an extension trait called UnicodeSegmentation. Exactly like std::io::Read whose read_to_string you called back in Lesson 6, a trait's methods are callable only when the trait is in scope. So bring it in at the top of lib.rs:

use std::fs;
use std::io::{self, Read};
use unicode_segmentation::UnicodeSegmentation;

Add the field to Counts. While you are in its doc comment, record what a grapheme is, and add a doc-test that shows the case where it differs from chars, using the combining "é" (e + U+0301):

/// A struct to hold counts of lines, words, bytes, characters, and graphemes.
///
/// `chars` counts Unicode scalar values (`char`); `graphemes` counts
/// user-perceived characters (grapheme clusters), so an accented letter
/// written as a base plus a combining mark counts as two `chars` but one
/// grapheme. `graphemes <= chars <= bytes` always holds.
///
/// ```
/// use rwc::Counts;
/// // ... the ascii assertions from Lesson 8 stay ...
/// assert_eq!(counts.graphemes, 26); // pure ASCII: one grapheme per char
///
/// // "e" + U+0301 (combining acute) renders as "é": 3 bytes, 2 chars, 1 grapheme.
/// let combined = Counts::from_text("e\u{301}");
/// assert_eq!(combined.bytes, 3);
/// assert_eq!(combined.chars, 2);
/// assert_eq!(combined.graphemes, 1);
/// ```
#[derive(Debug, Default, Clone, Copy, PartialEq, Eq)]
pub struct Counts {
    pub lines: usize,
    pub words: usize,
    pub bytes: usize,
    pub chars: usize,
    pub graphemes: usize,
}

Fill the field in from_text. This is the crate's method at work: graphemes(true) asks for extended grapheme clusters, the Unicode-correct default:

    pub fn from_text(text: &str) -> Counts {
        Counts {
            lines: text.lines().count(),
            words: text.split_whitespace().count(),
            bytes: text.len(),
            chars: text.chars().count(),
            graphemes: text.graphemes(true).count(),
        }
    }

Step 5 — Pin the toolchain: rust-toolchain.toml

Cargo.lock pins your dependencies, but not the compiler. Build rwc with a different rustc and you could get different warnings, different generated code, even a different result if a language feature changed. A rust-toolchain.toml at the project root pins the compiler too: rustup reads it and switches to exactly that toolchain whenever you run cargo in this directory. Reproducible dependencies and a reproducible compiler is what makes the whole build reproducible.

Find the version you are on:

rustc --version

Optional extensions

  1. Break the lock, then fix it. Change Cargo.toml to unicode-segmentation = "1.0" and run cargo build; watch it pick the newest 1.x and rewrite Cargo.lock. Then set it back to "1.13.3". Confirm with cargo tree which version you end up on and why.
  2. Try an incompatible bump. Set the requirement to "2" and run cargo build. Read the error: there is no 2.x, so resolution fails. This is semver refusing to hand you a version the requirement forbids.
  3. A harder grapheme. Add a file containing a single family emoji (e.g. 👨‍👩‍👧) and count it. Predict bytes, chars, and graphemes before you run; the grapheme count should be 1 while chars is much larger. Explain the gap.
  4. Offline build. After one successful build, run cargo build --offline. It works because the crate is cached and the lockfile already names it, so no network is needed. That is reproducibility paying off.

Check your understanding

Cargo.toml has unicode-segmentation = "1.13.3". Which versions may Cargo resolve to?

Right. "1.13.3" means ^1.13.3, i.e. >=1.13.3, <2.0.0. Cargo may pick a newer compatible 1.x, but 2.0.0 is a breaking major release and is excluded. To pick up a newer 1.x you run cargo update.

A bare version means neither "exactly this" nor "anything goes." It is a caret requirement: "1.13.3" = ^1.13.3 = >=1.13.3, <2.0.0. Compatible 1.x updates are allowed; a 2.0.0 major bump is not.

Why commit Cargo.lock to version control for rwc?

Right. The lockfile pins one exact version (and checksum) per crate, so a build reproduces identically across machines and time. Cargo can regenerate a lock from Cargo.toml, but then everyone might resolve to different versions.

Cargo can regenerate the lock from Cargo.toml and download crates fine without a committed lock, but then different machines may resolve to different versions. Committing it guarantees identical, reproducible builds. It has nothing to do with tracking source changes; that is git's job.

Cargo.lock pins your dependency versions. What does rust-toolchain.toml pin?

Right. rust-toolchain.toml pins rustc itself, so everyone compiles with the same compiler. Dependencies are the lockfile's job; the edition is the edition field in Cargo.toml. Reproducible build = pinned deps + pinned compiler.

Dependencies are pinned by Cargo.lock, and the edition is set by edition in Cargo.toml. rust-toolchain.toml pins a different thing: the compiler version rustup uses in this directory.
