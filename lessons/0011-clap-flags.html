Flags with clap (derive)

Lesson 10 gave rwc its first dependency. This lesson adds its second, clap, the standard crate for command-line parsing, and uses it to let the user pick which counts to print. Right now rwc prints all five columns every time; real tools like wc let you ask for just what you want (wc -l for lines, -w for words). You describe the command line as a struct with #[derive(Parser)] (the same struct-plus-derive pattern you used for Counts) and clap generates the parser, the --help, the --version, and the input validation from the field types. The anchoring feature: --lines/--words/--bytes/--chars/--graphemes flags that select columns.

Why clap, why now

Your argument handling so far is hand-rolled: env::args().skip(1).collect() feeds Source::from_args, which only tells files from - and rejects anything else starting with a dash. Add real flags to that by hand and it grows fast: you would have to parse -l, combined short flags like -lw, long forms like --words, print a --help screen, emit a usage error on a typo, and support --version. clap does all of it, and its derive API is something you can already read:

  • You describe the command line as a struct and put #[derive(Parser)] on it, the same mechanism that gives Counts its Debug and Copy.
  • Each field is one argument. A bool field is a flag; a Vec<String> field collects positional arguments.
  • The doc comment on each field becomes its help text. clap generates the parser, --help, and validation from the types.

Step 1 — Add clap with a feature

clap ships most of its machinery behind features: optional, additive parts of a crate you turn on in Cargo.toml. The derive API (the #[derive(Parser)] macro) lives behind the derive feature, so you ask for it explicitly:

cargo add clap --features derive

             Features:
             + std
             - derive     # (now enabled by --features)
      Adding anstyle v1.0.14
      Adding clap_builder v4.6.0
      Adding clap_derive v4.6.1
      ... # 20 packages in total

The entry clap writes looks different from Lesson 10's. A dependency with features is a table, not a bare string:

[dependencies]
clap = { version = "4.6.1", features = ["derive"] }
unicode-segmentation = "1.13.3"

The version still follows the caret rule from Lesson 10 ("4.6.1" means >=4.6.1, <5.0.0); the features array is what turns on the derive macro. Notice the terminal said it added 20 packages, not one. clap depends on other crates, which depend on still others: transitive dependencies. cargo tree shows the whole subtree one cargo add brought in:

Step 2 — Describe the CLI as a struct

Bring the derive macro into scope and describe the whole command line as one struct. Put this at the top of main.rs. The #[command(...)] attribute configures the program (version reads your Cargo.toml version; about uses the struct's doc comment); each field's #[arg(...)] configures one argument.

use clap::Parser;
use rwc::{Counts, Report, Source};
use std::env;
use std::error::Error;

The #[arg(short, long)] on a bool field makes a flag with both a short form (-l, the first letter of the field) and a long form (--lines). When the first letter clashes or isn't the conventional one (wc uses -c for bytes and -m for chars), you set the short letter explicitly with short = 'c'.

// main.rs — add this above `fn main`
/// Count lines, words, bytes, characters, and graphemes.
///
/// With no count flags, every column is shown. Pass one or more flags to
/// print only those columns.
#[derive(Parser)]
#[command(version, about, long_about = None)]
struct Cli {
    /// Show the line count
    #[arg(short, long)]
    lines: bool,

    /// Show the word count
    #[arg(short, long)]
    words: bool,

    /// Show the byte count
    #[arg(short = 'c', long)]
    bytes: bool,

    // TODO (your turn): add `chars` (short 'm') and `graphemes` (short 'g'),
    // each a `bool` flag with a `///` doc comment, mirroring `bytes`

    /// Files to count; omit or use "-" for standard input
    paths: Vec<String>,
}

The final field, paths, has no #[arg] attribute and is not a bool, so clap treats it as a positional argument, and because it is a Vec it collects all the leftover non-flag arguments (the file names). That is exactly the Vec<String> your Source::from_args already expects.

Step 3 — Select columns in the library

The flags decide which columns print. Keep that decision in the library, as a small type, so main stays thin and the formatting stays in Report. Add a Columns struct near Report in lib.rs: five booleans and two constructors. all() turns everything on; from_flags maps the CLI flags, defaulting to all() when the user passes none.

Step 4 — Wire clap into main

Now replace the hand-rolled parsing. Cli::parse() reads the process arguments, applies everything the struct declared, and, on --help, --version, or a bad argument, prints and exits for you. The old env::args line and its use std::env; go away; import Columns from the library.

use clap::Parser;
use rwc::{Columns, Counts, Report, Source};
use std::error::Error;

Step 5 — What clap gave you for free

You wrote a struct; clap generated a whole CLI. --help is built from the #[command] about text and every field's doc comment:

cargo run -- --help
Count lines, words, bytes, characters, and graphemes

Usage: rwc [OPTIONS] [PATHS]...

Arguments:
  [PATHS]...  Files to count; omit or use "-" for standard input

Options:
  -l, --lines      Show the line count
  -w, --words      Show the word count
  -c, --bytes      Show the byte count
  -m, --chars      Show the character count (Unicode scalar values)
  -g, --graphemes  Show the grapheme count (user-perceived characters)
  -h, --help       Print help
  -V, --version    Print version

--version comes from #[command(version)], which reads Cargo.toml:

cargo run -- --version
rwc 0.1.0

And an unknown flag is rejected with a usage message on stderr and a non-zero exit code, validation you did not write:

cargo run -- --nope demo.txt
error: unexpected argument '--nope' found

  tip: to pass '--nope' as a value, use '-- --nope'

Usage: rwc [OPTIONS] [PATHS]...

For more information, try '--help'.

Note clap added -h/--help and -V/--version itself. That is why version uses a capital -V, leaving lowercase -v free for a future --verbose.

Optional extensions

  1. Add --verbose. Add a #[arg(short, long)] verbose: bool field and print the resolved Columns to stderr when it is set. Notice clap gives it -v automatically, the letter it kept free by using -V for version.
  2. Header row. Add a --header flag that prints a column-title row (lines words …) matching the selected columns before the counts. Reuse the same Columns to decide which titles to show.
  3. Read the generated help harder. Run cargo run -- -h vs --help and compare; add a #[command(long_about = "…")] and see where the longer text appears. The attributes drive all of it; you write no output code of your own.
  4. Trim the tree. Run cargo tree -e no-dev and cargo tree -d (duplicates). clap is a big dependency; some projects use lighter parsers (lexopt, pico-args) when binary size matters. Weigh the trade-off.

Check your understanding

In the Cli struct, why does the paths field become a positional argument while lines becomes a flag?

Right. clap reads the type and attributes. A bool with #[arg(short, long)] is a flag; a field without short/long is positional, and a Vec collects all the leftover values. Neither name nor position is what decides it.

It is the type and attributes, not the name or ordering. #[arg(short, long)] on a bool makes a flag; a field with no short/long is positional, and a Vec<String> gathers every leftover argument.

You ran cargo add clap --features derive and cargo tree shows 20-plus extra crates. What are those?

Right. A dependency's own dependencies come along transitively. One cargo add can pull in a whole subtree; cargo tree shows it, and Cargo.lock pins every crate in it.

They are transitive dependencies: the crates clap itself depends on (and theirs, in turn). They are not stray features or per-platform copies; cargo tree shows the real subtree that one cargo add brought in.

Where does the text in rwc --help come from?

Right. The derive macro turns your doc comments and #[command]/#[arg] attributes into the help screen (and --version, and the usage errors). You wrote none of that output code.

You don't write any help-printing code. clap's derive macro builds --help from the struct's doc comments and #[command]/#[arg] attributes at compile time.
