Tests & the feedback loop

Make rwc's counting testable, then write a test suite that cargo test runs — so the next refactor can't silently break it.

Lesson goals

By the end of this lesson you will be able to:

  • Extract a pure function so its behaviour can be tested without file I/O.
  • Write unit tests in a #[cfg(test)] module using #[test] and the assert_eq! / assert! macros.
  • Run the suite with cargo test and read its pass/fail report.
  • Explain why these tests become a safety net for the refactoring in Module 2.
Starting point Work in 0003-rwc/, which already contains the finished tool from Lesson 2. As before, the lines highlighted in green are the ones to add or change.

Why now, and why a pure function first

Right now every piece of rwc lives inside main: it reads an argument, reads a file, counts, and prints. That is hard to test, because to check the counting you would have to run the whole program against a real file and inspect its printed output. The fix is to separate the part you want to verify — the counting logic — from the parts that touch the outside world (arguments, files, the screen).

A pure function — one that just maps inputs to outputs with no I/O — is the easiest thing in any language to test: you hand it a value and assert on what it returns. So before writing a single test, we extract the counting into its own function.

Step 1 — Extract a testable counts function

Pull the three counting lines out of main into a function that takes the text and returns the three totals, then call it from main:

use std::env;
use std::fs;
use std::error::Error;

fn counts(text: &str) -> (usize, usize, usize) {
    let lines = text.lines().count();
    let words = text.split_whitespace().count();
    let chars = text.chars().count();
    (lines, words, chars)
}

fn main() -> Result<(), Box<dyn Error>> {
    let path = env::args().nth(1).ok_or("usage: rwc <file>")?;
    let text = fs::read_to_string(&path)?;
    let (lines, words, chars) = counts(&text);
    println!("{lines:>6} {words:>6} {chars:>6} {path}");
    Ok(())
}

What changed, and why each piece looks the way it does:

  • fn counts(text: &str) -> (usize, usize, usize) — A new function. It takes text as a &str — a borrowed view of the string (the counting only needs to read the text, not own it), which is the same borrowing idea you met as &path in Lesson 1. It returns a tuple of three usize values: a tuple is a fixed-size group of values, written in parentheses, and usize is the unsigned integer type Rust uses for counts and sizes. In short: pass in text, get back its line, word, and character counts.
  • The three let lines and (lines, words, chars) — These moved verbatim out of main. The final line is the function's return value: Rust returns the value of the last expression in a function when it has no trailing semicolon, so this bare tuple is what counts hands back. No return keyword is needed.
  • let (lines, words, chars) = counts(&text); — Back in main, this calls the function and destructures the returned tuple into three named variables in one step, so the println! below it works exactly as before. We pass &text — a borrow — so main keeps ownership of the file contents.

Confirm it still behaves identically before adding any tests:

cargo run -- ../MISSION.md   # same counts as Lesson 2

Step 2 — Add a test module

At the bottom of src/main.rs, add a module of tests:

fn main() -> Result<(), Box<dyn Error>> {
    // ... unchanged ...
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn counts_ascii() {
        let (l, w, c) = counts("hello world\nsecond line\n");
        assert_eq!(l, 2);
        assert_eq!(w, 4);
        assert_eq!(c, 24);
    }

    #[test]
    fn counts_are_unicode_aware() {
        let text = "வணக்கம்\n";
        let (l, w, c) = counts(text);
        assert_eq!(l, 1);
        assert_eq!(w, 1);
        assert!(c < text.len(), "chars ({c}) should be fewer than bytes ({})", text.len());
    }

    #[test]
    fn empty_input() {
        assert_eq!(counts(""), (0, 0, 0));
    }
}

What each annotation and macro does:

  • #[cfg(test)] — An attribute (the #[…] syntax) that conditionally compiles the module: the code inside is built only when you run cargo test, and is left out of your normal release binary entirely. So tests cost you nothing in the shipped program. (Rust Book, Ch. 11.1.)
  • mod tests { … } — Declares a module named tests to hold the test functions. The name is conventional, not required.
  • use super::*; — super means the parent module (here, the file's top level), and * brings everything from it into scope. This is what lets the tests call counts without qualifying it — the test module is nested, so it can't see counts until you import it.
  • #[test] — Marks a function as a test. cargo test builds a special test runner that finds every #[test] function, runs it, and reports whether it passed (returned normally) or failed (panicked).
  • assert_eq!(left, right) — Passes if the two values are equal; if not, it panics and prints both values, so a failure tells you exactly what you got versus what you expected. (assert!(cond, "msg…") checks a boolean and lets you add a custom failure message, as in the Unicode test.)
  • The three tests — verify ordinary ASCII counts; that counting is Unicode-aware (the Tamil string is one word but has fewer characters than bytes — the .chars() vs .len() point from Lesson 1, now pinned down by a test); and the empty-input edge case.

Step 3 — Run the suite

cargo test

You should see all three pass:

running 3 tests
test tests::empty_input ... ok
test tests::counts_ascii ... ok
test tests::counts_are_unicode_aware ... ok

test result: ok. 3 passed; 0 failed; 0 ignored; ...

Now make a test fail on purpose — change assert_eq!(w, 4) to assert_eq!(w, 5) and re-run. Read how cargo test reports the mismatch (it prints left and right), then change it back. That failure report is the feedback loop you will use for every change from here on.

Outcome rwc now has a pure counts function covered by a runnable test suite. cargo test gives you an instant, automatic check — so when Module 2 refactors the program's structure, any change that breaks the counting will fail a test immediately instead of slipping through.

Optional extensions

  1. Add a case. Write a test for text with no trailing newline (e.g. "one two") and confirm the line count is what you expect.
  2. Test the binary end to end. Rust's tests/ directory holds integration tests. Testing a command-line binary's full behaviour (arguments, exit codes, stdout) is usually done by running the compiled program as a subprocess — the assert_cmd crate is the standard tool. That needs a dependency, which is Module 3 territory; note it as a goal.
  3. Document example. Add a /// doc comment with a code example above counts; cargo test runs those examples as "doc-tests" too.

Check your understanding

What does the #[cfg(test)] attribute do?

Right — the module is built only under `cargo test`, so tests add nothing to the normal release binary. (#[test] is what marks each function.)

That's #[test]'s job. #[cfg(test)] conditionally compiles the whole module — only when running tests, never in the release binary.

Why extract counts into its own function before testing?

Yes — isolating the logic from file/argument/print I/O lets a test pass input and assert on output directly, with no files involved.

It's about testability, not speed: a pure function (no I/O) can be checked by passing input and asserting the return value, unlike logic buried in main.

When does assert_eq! cause a test to fail?

Correct — unequal values make assert_eq! panic and print both sides; a panic in a #[test] function is reported as a failure.

assert_eq! fails specifically when left != right (it then panics and prints both values). Zeroes and tuples are fine if they match.
