Rust · Lesson 5 · Diagnostics

Reading borrow-checker errors

Lesson 4 gave you the rules; this lesson is about what happens when you break them. You will deliberately introduce four classic E03xx errors into rwc, learn to read each message part by part, and apply its standard fix.

Lesson goals

By the end of this lesson you will be able to:

  • Read a rustc error structurally: find the error code, the primary span, the secondary labels, and the help in any message.
  • Recognise the four most common borrow errors on sight (E0596, E0382, E0502, E0499) and apply each one's standard fix.
  • Explain why each error is the compiler enforcing a Lesson 4 rule, not being pedantic.
  • Use rustc --explain and the online error index to decode any error you haven't seen before.
Starting point Work in 0005-rwc/, a clean copy of the finished, tested tool from Lesson 4 (multi-file counting with a grand total). Confirm it is green before you start: cargo test should report 4 passed. Each step below has you break the code on purpose, read the resulting error, then put it back. Nothing here changes rwc permanently; you are practising reading errors, not adding a feature.

Why a lesson on errors

In C, a type error is often a terse one-liner and a data race produces no error at all, just a crash later in production. Rust does the opposite: it refuses to compile a whole category of bugs, and it spends real effort explaining each refusal. A borrow-checker error tells you, with line numbers and arrows, exactly which Lesson 4 rule the code violates and usually how to fix it. This lesson teaches you to read that explanation quickly instead of guessing, so the borrow checker catches your mistakes before they ship rather than slowing you down.

Every error below is one you will hit for real while writing Rust. We provoke them in rwc on purpose so you meet them in code you already understand.

How to read any rustc error

Every rustc diagnostic has the same structure. Learn the parts once and you can skim any future error:

  • error[E0382]: borrow of moved value gives the error code (a concept you can look up) and a one-line summary. The code is the most searchable thing in the whole message.
  • --> src/main.rs:37:8 marks the location: file, line, column of the primary problem.
  • The ^^^^^ caret line is the primary span: the exact code the compiler is complaining about, underlined with carets.
  • The ----- underlines with text are the secondary labels: other spots that explain the primary one (where a value was moved, where a borrow started, where it is "later used"). For borrow errors these are the whole story: they show you the two events that conflict.
  • help: and note: give a suggested fix (often the literal edit to make) and extra context. The help is right far more often than not.

Reading order that works: code → summary → the two labelled spans → help. Borrow errors are almost always a conflict between two points in the code, and the labels name both.

Step 1 — E0596: a value you forgot to make mutable

The most common borrow error is also the simplest. In main, drop the mut from the totals binding so it reads:

    let total = (0, 0, 0);
    for path in &paths {

Run cargo build. You get:

error[E0596]: cannot borrow `total` as mutable, as it is not declared as mutable
  --> src/main.rs:35:18
   |
35 |         add_into(&mut total, (lines, words, chars));
   |                  ^^^^^^^^^^ cannot borrow as mutable
   |
help: consider changing this to be mutable
   |
29 |     let mut total = (0, 0, 0);
   |         +++

For more information about this error, try `rustc --explain E0596`.

Decoding it, part by part:

  • The primary span (line 35) points at &mut total, the place you asked for a mutable borrow. That is the action the compiler can't allow.
  • The reason is in the summary: total "is not declared as mutable." In Rust every binding is immutable by default (the opposite of C and Go), and you can only hand out a &mut to something declared mut. The mismatch is between the request on line 35 and the declaration on line 29.
  • The help shows the exact fix, adding mut at line 29, and even draws the +++ where the three new characters go.

Fix: put the mut back. This is the compiler enforcing that mutation is never silent: a reader (and the compiler) can tell a binding is mutable just by looking at its declaration.

Step 2 — E0382: using a value after it moved

This is the central ownership error, and the reason Lesson 4 used for path in &paths instead of for path in paths. Remove that & so the loop consumes the vector:

    let mut total = (0, 0, 0);
    for path in paths {
        let text = fs::read_to_string(&path)

(You also need &path on the next line now, because path is an owned String rather than a reference, but that is not what breaks.) Build it:

error[E0382]: borrow of moved value: `paths`
   --> src/main.rs:37:8
    |
 23 |     let paths: Vec<String> = env::args().skip(1).collect();
    |         ----- move occurs because `paths` has type `Vec<String>`, which does not implement the `Copy` trait
...
 30 |     for path in paths {
    |                 ----- `paths` moved due to this implicit call to `.into_iter()`
...
 37 |     if paths.len() > 1 {
    |        ^^^^^ value borrowed here after move
    |
note: `into_iter` takes ownership of the receiver `self`, which moves `paths`
help: consider iterating over a slice of the `Vec<String>`'s content to avoid moving into the `for` loop
    |
 30 |     for path in &paths {
    |                 +

For more information about this error, try `rustc --explain E0382`.

This message has three labelled spans, and reading all three is the whole skill. In order:

  • Line 23: "move occurs because paths has type Vec<String>, which does not implement the Copy trait." This is the root cause. A Vec owns heap memory, so it is a move type, not a Copy type: handing it somewhere transfers ownership rather than duplicating it. (The rule from Lesson 4: own a resource ⇒ move; trivially copyable ⇒ Copy.)
  • Line 30: "paths moved due to this implicit call to .into_iter()." This is the subtle part the message points out: for x in collection desugars to collection.into_iter(), and into_iter takes ownership (the note spells this out). So the loop didn't borrow paths; it consumed it. After the loop, paths no longer exists.
  • Line 37: "value borrowed here after move." This is the primary span: the place that fails. Calling paths.len() needs paths to still be alive, but line 30 already moved it away. The conflict is between line 30 (move) and line 37 (use).

Fix: do what the help says: iterate over &paths, which calls .iter() and only borrows the vector, leaving it intact for paths.len() afterwards. That is why Lesson 4 wrote the loop that way. Put the & back (and restore read_to_string(path)).

Step 3 — E0502: a shared and a mutable borrow at the same time

Now provoke the main rule from Lesson 4: shared XOR mutable. Inside the loop, take a plain (shared) borrow of total, then mutate it through add_into, then use the shared borrow afterwards:

        println!("{lines:>6} {words:>6} {chars:>6} {path}");
        let peek = &total;
        add_into(&mut total, (lines, words, chars));
        println!("running subtotal so far: {:?}", peek);
error[E0502]: cannot borrow `total` as mutable because it is also borrowed as immutable
  --> src/main.rs:36:18
   |
35 |         let peek = &total;
   |                    ------ immutable borrow occurs here
36 |         add_into(&mut total, (lines, words, chars));
   |                  ^^^^^^^^^^ mutable borrow occurs here
37 |         println!("running subtotal so far: {:?}", peek);
   |                                                   ---- immutable borrow later used here

For more information about this error, try `rustc --explain E0502`.

The three labels are the rule, made concrete:

  • Line 35: "immutable borrow occurs here." peek = &total is a shared & borrow.
  • Line 36: "mutable borrow occurs here." &mut total is an exclusive borrow, and exclusive means nothing else may be borrowing total right now.
  • Line 37: "immutable borrow later used here." This is the key one. The shared borrow from line 35 is still alive on line 37 (you use peek there), so it overlaps the mutable borrow on line 36. Two borrows, one shared and one mutable, alive at the same instant: forbidden.

Notice why the third label matters so much: if you deleted line 37, the shared borrow would end right after line 35 (its last use), the overlap would vanish, and the code would compile. The compiler tracks a borrow's life from where it starts to its last use, not to the end of the block. This is "non-lexical lifetimes," and it is why reordering or removing a later use often fixes a borrow error.

Fix: remove the peek lines. If you genuinely needed to read the running subtotal, you would read it before the mutation or after it (a fresh borrow each time), rather than holding one across the other.

Step 4 — E0499: two mutable borrows at once

The mutable side of the same rule: &mut is exclusive even against another &mut. Force two live mutable borrows of total:

        println!("{lines:>6} {words:>6} {chars:>6} {path}");
        let a = &mut total;
        let b = &mut total;
        add_into(b, (lines, words, chars));
        add_into(a, (0, 0, 0));
error[E0499]: cannot borrow `total` as mutable more than once at a time
  --> src/main.rs:36:17
   |
35 |         let a = &mut total;
   |                 ---------- first mutable borrow occurs here
36 |         let b = &mut total;
   |                 ^^^^^^^^^^ second mutable borrow occurs here
37 |         add_into(b, (lines, words, chars));
38 |         add_into(a, (0, 0, 0));
   |                  - first borrow later used here

For more information about this error, try `rustc --explain E0499`.
  • Line 35 creates the first &mut (a); line 36 creates a second (b), the primary span, because the second is the one that can't exist.
  • Line 38: "first borrow later used here." Same non-lexical-lifetime logic: because a is used on line 38, its borrow is still alive when b is created on line 36. Two exclusive borrows, overlapping. Had you finished using a before creating b, there would be no overlap and no error.

Fix: remove the a/b lines and keep the original single add_into(&mut total, …). One exclusive borrow, used and released, then the next, which is exactly what the loop already did.

Tool · rustc --explain

Every error message ends with try `rustc --explain E0XXX`. Run it for a full prose write-up of the error with a minimal example and the reasoning behind it. It works offline and is often clearer than searching the web:

rustc --explain E0502

The same content lives online in the Rust Error Index (linked at the foot of this page). When you hit an E03xx you don't recognise, that is the fastest authoritative source.

The four, at a glance

Almost every borrow error you meet early on is one of these. The pattern to internalise: find the two conflicting spans, identify which rule they break, apply the fix.

  • E0596: cannot borrow as mutable, not declared mutable. You took &mut of an immutable binding. Fix: add mut to the let.
  • E0382: use of moved value. You used a value after it was moved (often by passing it by value or by for x in collection). Fix: borrow instead of move (&), or .clone() if you truly need a second owner.
  • E0502: mutable borrow while an immutable borrow is live. Shared and mutable overlapped. Fix: end the shared borrow before mutating (don't use it across the mutation).
  • E0499: two mutable borrows at once. Two exclusive borrows overlapped. Fix: finish with the first before taking the second.

The common thread under all four: a borrow lives until its last use, and the rule is shared XOR mutable. Most fixes are just shrinking a borrow's life so two of them stop overlapping.

Outcome You can now read a borrow-checker error part by part: code, summary, the conflicting spans, help. You met the four you will hit most (E0596, E0382, E0502, E0499), saw each as one of Lesson 4's rules being enforced, and learned the model that fixes most of them: a borrow lasts until its last use, and shared excludes mutable. rustc --explain and the Error Index cover everything else.

Optional extensions

  1. Provoke E0505 (move out of a borrowed value). Inside the loop add let r = &total;, then on a later line write let stolen = total; and use r after it. Read how the message differs from E0502: here you are moving the value out while it is borrowed, not borrowing it again.
  2. Fix E0382 the other way. In Step 2, instead of restoring the &, leave for path in paths and replace the later paths.len() with a length captured before the loop (let n = paths.len();). Confirm it compiles, then decide which fix you prefer and why: borrowing vs. reading the length up front.
  3. Run rustc --explain on all four. Compare its canonical minimal example for each error to the rwc version you just produced. Seeing the same error in two different shapes is what makes it recognisable next time.

Check your understanding

In a borrow error, what does a secondary label like "first borrow later used here" tell you?

Right. A borrow lives until its last use. "Later used here" marks that last use, which proves the borrow is still alive and therefore overlaps the other borrow. Shrinking that life (using it earlier, or not at all) is often the fix.

It marks where the borrow is last used, which proves it is still alive at that point, and therefore overlaps the conflicting borrow. It is not where the borrow started, and deleting the line is only sometimes the right fix.

Why does for path in paths (no &) cause E0382 when you use paths afterwards?

Correct. for x in collection desugars to collection.into_iter(), which takes ownership. After the loop, paths has been moved away, so any later use is a use-after-move. Iterating &paths borrows instead.

It is about ownership: for x in collection desugars to .into_iter(), which moves the vector into the loop. The vector is fine to iterate many times by reference; the problem is that by-value iteration consumed it.

You wrote add_into(&mut total, …) and got E0596 ("not declared as mutable"). The fix is:

Right. You can only take a &mut of a binding declared mut. The error even shows the exact edit (+++ before total on the let line).

The request for &mut is fine; the binding just isn't mutable. Add mut to its let. Passing by value would defeat the in-place accumulation, and a second borrow makes things worse, not better.
