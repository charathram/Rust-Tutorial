Strings & slices: String vs &str

You have used String and &str since Lesson 1 without naming the difference. Now learn what a Rust string actually is — UTF-8 bytes — and teach rwc to count characters as well as bytes. Along the way: owned vs borrowed text, how slicing works, and why a byte index can land in the middle of a character.

Lesson goals

By the end of this lesson you will be able to:

  • Tell String (owned) from &str (a borrowed view), and choose between them deliberately.
  • Explain that a &str is UTF-8 bytes: .len() counts bytes, .chars().count() counts characters.
  • Slice a string with &s[a..b], know it yields a &str, and know why an off-boundary index panics.
  • Add a character count to rwc alongside its byte count.
  • Write a function that returns a borrowed slice safely.

Why strings, why now

rwc already moves between two string types on every run. Source::read hands back a String — the whole file, owned. Counts::from_text takes a &str — it only reads the text, so it borrows instead of copying. You wrote both signatures in Lesson 6 and they just worked; this lesson explains the distinction so you can make it on purpose.

There is also unfinished business. Lesson 6 had rwc count bytes (text.len()). On the ASCII test files that looked like a character count — but it isn't, and the moment a file holds an accented letter or any non-Latin script the two numbers part ways. Real wc draws the same line: -c counts bytes, -m counts characters. This lesson explains why they differ and adds a real character count, so rwc reports both.

Step 1 — String vs &str: owned vs borrowed

Two types, one job — holding text — but they sit on opposite sides of ownership:

The pair

String — an owned, growable, heap-allocated buffer of UTF-8 bytes. It can be appended to, it owns its memory, and it frees that memory when it drops. This is what you get from fs::read_to_string or format!.

&str (a "string slice") — a borrowed view into UTF-8 text someone else owns: under the hood just a pointer and a length. It allocates nothing and owns nothing. A string literal like "-" is a &str (it borrows from the program binary); so is &some_string.

It is the same owned-vs-borrowed split you learned for references in Lesson 4, specialized to text. And you can see it already in rwc's own signatures:

  • fn read(&self) -> io::Result<String> — returns an owned String, because the file's contents must live on after the call.
  • fn from_text(text: &str) -> Counts — borrows the text; it counts and returns, never needing to own a copy.
  • fn name(&self) -> &str — returns a slice borrowed out of self: no new string, just a view into the path already stored in the Source.
  • fn describe(&self) -> String — returns an owned String, because it builds new text with format! that nothing else owns.

The rule of thumb that follows:

  • Take &str as a parameter. A caller holding a String passes &s; a caller with a literal passes it directly. That is why from_text(&text) works even though text is a String — a &String coerces to &str. One parameter type accepts both.
  • Return &str when handing back a view into data the caller already owns (like name), and return String when you produce new text (like describe).

Step 2 — A &str is bytes, not characters

Here is the fact that everything else in this lesson rests on: a Rust string is a sequence of UTF-8 bytes, and .len() reports the number of bytes, not characters. For plain ASCII every character is one byte, so the two numbers are equal — which is exactly why Lesson 6's byte count looked like a character count on the ASCII fixtures. But UTF-8 encodes most of the world's characters in two to four bytes:

let s = "café";
s.len();             // 5  — bytes ('é' is encoded as two)
s.chars().count();   // 4  — characters (Unicode scalar values)

Laid out byte by byte, the mismatch is concrete:

"café"   c     a     f     é
bytes:  0x63  0x61  0x66  0xc3 0xa9     ← 5 bytes; 'é' takes two
chars:   1     2     3      4           ← 4 characters

.len() walks the raw bytes; .chars() decodes them back into chars (Unicode scalar values) and .count() tallies those. That is the whole of wc's -c (bytes) versus -m (characters) distinction — and the reason rwc should report both.

Step 4 — Slicing and UTF-8 boundaries

A slice expression &s[start..end] produces a &str that borrows the bytes from start up to (not including) end. The indices are byte offsets, and the slice must begin and end on a character boundary. Land inside a multi-byte character and it is not a compile error — it panics at runtime:

fn main() {
    let s = "café";
    let good = &s[0..3]; // "caf" — ends on a boundary
    let bad = &s[0..4];  // byte 4 lands inside 'é'
    println!("{good} {bad}");
}
thread 'main' panicked at src/main.rs:4:17:
end byte index 4 is not a char boundary; it is inside 'é' (bytes 3..5 of string)

The message is precise: byte 4 is the second byte of é (which occupies bytes 3 and 4), so it is not a place a &str may start or stop. Two safe ways to avoid the panic:

  • s.get(start..end) returns Option<&str> — None instead of a panic when the range is off-boundary. s.get(0..4) is None; s.get(0..3) is Some("caf").
  • Count by characters, not bytes. char_indices() yields each character paired with the byte offset where it begins — always a valid boundary:
for (i, c) in "café".char_indices() {
    println!("{i}: {c}");
}
// 0: c
// 1: a
// 2: f
// 3: é

Use that to take the first n characters safely. char_indices().nth(n) gives the byte offset where character n begins (the first index past the n you want); slice up to there. If the string has fewer than n characters there is no such offset, so return the whole thing.

    pub fn first_chars(s: &str, n: usize) -> &str {

Notice what the signature says: first_chars takes a &str and returns a &str that points into it. The returned slice borrows from s, so it cannot outlive it — the same borrowing rule from Lesson 4, now spanning a function boundary. Making that relationship explicit (with a lifetime) is exactly what Lesson 9 is about; for now the compiler infers it.

Outcome rwc now counts characters as well as bytes, and you can say exactly why those numbers differ. You learned that a Rust string is UTF-8 bytes, that String owns while &str borrows, that slicing works on byte offsets and must respect character boundaries, and you wrote a function that safely returns a borrowed slice — the borrow that Lesson 9 will name with a lifetime.

Optional extensions

  1. Graphemes vs scalar values. Even chars().count() isn't the "visible characters" a human sees: "e\u{301}" (an e plus a combining accent) is one grapheme but two chars. Add the unicode-segmentation crate and compare graphemes(true).count() with chars().count() on such a string.
  2. A non-panicking slice. Write safe_slice(s: &str, a: usize, b: usize) -> Option<&str> that wraps s.get(a..b), and a test proving an off-boundary range yields None rather than panicking.
  3. Preview long names. Use first_chars to truncate any source label longer than, say, 20 characters to first_chars(name, 20) plus an ellipsis — proving the helper handles a multi-byte label without splitting a character.

Check your understanding

What does "café".len() return?

Right — .len() is the byte length, and é encodes as two bytes, so "café" is 5 bytes. For the character count (4) you'd use .chars().count().

.len() reports bytes, not characters. é takes two UTF-8 bytes, so "café" is 5 bytes; .chars().count() would give 4.

&s[0..4] where s = "café" and byte 4 is inside é — what happens?

Right — slice indices are byte offsets that must fall on character boundaries; an off-boundary index is a runtime panic, not a compile error. Use s.get(0..4) for an Option instead.

It compiles fine — the index is only checked at runtime, where landing inside é panics. It never rounds; for a non-panicking version use s.get(0..4), which returns None.

A function builds a brand-new label with format! and returns it. What type should it return?

Right — the new text isn't owned by anyone else, so the function must own and return it as a String. You return &str only when handing back a view into data that already exists (like name).

Newly built text has no other owner to borrow from, so a &str would dangle. Return String when you create text; return &str only for a view into existing data.
