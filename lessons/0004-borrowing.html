Borrowing & references

Grow rwc to count several files and print a grand total. Passing data around without giving up ownership is what forces the two kinds of reference — shared & and mutable &mut — and the one rule that governs them.

Lesson goals

By the end of this lesson you will be able to:

  • Pass a value to a function by borrowing it (&) instead of moving it, and say why that keeps the caller's ownership intact.
  • Hand out a mutable borrow (&mut) so a function can modify a value in place.
  • State the borrowing rule — either one &mut or any number of &, never both — and explain what it buys versus C and Go.
  • Iterate over a collection by reference so it remains usable afterwards.
Starting point Work in 0004-rwc/, which already contains the finished, tested tool from Lesson 3. As before, the lines highlighted in green are the ones to add or change in each step.

Why borrowing, and why now

You have already used a borrow without dwelling on it: counts(&text) hands the counting function a reference to the text rather than the text itself, so main keeps ownership and can still use text afterwards. That was the cheap, obvious case. This lesson makes rwc accept several files and print a total across them — and that feature forces the rest of the borrowing model into the open: you will keep a list of paths alive while looping over it, and you will let a helper function modify a running total in place. Each need maps onto a specific kind of reference, and Rust enforces exactly one rule about how they may coexist.

A reference is Rust's way of letting code use a value without taking ownership of it — no move, no copy, just temporary access that the compiler tracks. Coming from C, a reference is morally a pointer the compiler refuses to let you misuse; coming from Go, it is like passing a pointer except the aliasing rules are checked at compile time instead of trusted at runtime.

Step 1 — Accept several file paths

Replace the single-argument handling in main with code that gathers all the path arguments and loops over them:

fn main() -> Result<(), Box<dyn Error>> {
    let paths: Vec<String> = env::args().skip(1).collect();
    if paths.is_empty() {
        eprintln!("usage: rwc <file>...");
        std::process::exit(2);
    }

    for path in &paths {
        let text = fs::read_to_string(path).map_err(|e| format!("{path}: {e}"))?;
        let (lines, words, chars) = counts(&text);
        println!("{lines:>6} {words:>6} {chars:>6} {path}");
    }
    Ok(())
}

What each new piece does, and why it is written this way:

  • let paths: Vec<String> = env::args().skip(1).collect(); — Three things at once. env::args() yields the command-line arguments as an iterator, and the first item is always the program's own name, so .skip(1) drops it and leaves just the user's file paths. .collect() then drains that iterator into a collection — but collect can build many different collections, so it needs to be told which one; the type annotation : Vec<String> on the left is what tells it. A Vec<String> is a growable, heap-allocated array (a "vector") of owned String values — Rust's equivalent of a dynamic list, and the owner of every path it holds.
  • if paths.is_empty() { … std::process::exit(2); } — With multiple files now allowed, "no argument given" is the empty-vector case. is_empty() returns true when there are zero paths; we print usage to stderr and exit with code 2, exactly as the single-file version did.
  • for path in &paths { — The central borrowing point of this step. The & means we loop over a shared borrow of the vector, so each path is a &String (a reference into paths) and the vector itself is only lent to the loop, not consumed by it. That is why paths is still alive and usable after the loop — we rely on that in Step 2 to ask paths.len(). Had you written for path in paths (no &), the loop would have taken ownership of the vector and consumed it, and paths would no longer exist afterwards.
  • fs::read_to_string(path).map_err(|e| format!("{path}: {e}"))? — The error message now leads with which file failed ({path}), which matters once several files are in play. The ? still bubbles the first failure out of main as before.

The counts(&text) and println! lines are unchanged — they have simply moved inside the loop so they run once per file.

Step 2 — Total across files with a &mut accumulator

To print a grand total we need a running tally that each file adds into. Add a small function that takes the total by mutable reference and updates it, then call it inside the loop and print the total at the end:

fn counts(text: &str) -> (usize, usize, usize) {
    // ... unchanged ...
}

fn add_into(total: &mut (usize, usize, usize), c: (usize, usize, usize)) {
    total.0 += c.0;
    total.1 += c.1;
    total.2 += c.2;
}

fn main() -> Result<(), Box<dyn Error>> {
    let paths: Vec<String> = env::args().skip(1).collect();
    if paths.is_empty() {
        eprintln!("usage: rwc <file>...");
        std::process::exit(2);
    }

    let mut total = (0, 0, 0);
    for path in &paths {
        let text = fs::read_to_string(path).map_err(|e| format!("{path}: {e}"))?;
        let (lines, words, chars) = counts(&text);
        println!("{lines:>6} {words:>6} {chars:>6} {path}");
        add_into(&mut total, (lines, words, chars));
    }

    if paths.len() > 1 {
        let (lines, words, chars) = total;
        println!("{lines:>6} {words:>6} {chars:>6} total");
    }
    Ok(())
}

The new borrowing ideas here:

  • fn add_into(total: &mut (usize, usize, usize), c: (usize, usize, usize)) — The first parameter is a mutable borrow: &mut lends the function the totals tuple with permission to read and write it. While that borrow is held, the compiler guarantees nothing else can touch total — the borrow is exclusive. The second parameter, c, is passed by value with no & at all: a tuple of three integers is a Copy type (cheap to duplicate bit-for-bit), so there is nothing to be gained by borrowing it. A String in that position would move, and you would borrow it to avoid that — choosing by-value vs by-reference is a judgement you will make constantly.
  • total.0 += c.0; — Tuple fields are reached by position: .0, .1, .2. += adds in place. Because total arrived as &mut, writing through it actually updates the caller's value.
  • let mut total = (0, 0, 0); — The mut keyword marks the binding as mutable; without it the compiler forbids the in-place updates. (Rust bindings are immutable by default — the opposite of C and Go.)
  • add_into(&mut total, (lines, words, chars)); — The caller must spell out &mut too. Unlike C, where you cannot tell at a call site whether a function will mutate what you pass, Rust makes the exclusive loan visible right there in the call.
  • if paths.len() > 1 { … } — Only print a total line when there was more than one file (matching the behaviour of the real wc). This reads paths.len() — which only works because Step 1 iterated with &paths and so left the vector intact.
Concept · the borrowing rule

A reference gives temporary access to a value the caller still owns. There are exactly two kinds, and one rule governs how they coexist:

  • &T — a shared (immutable) borrow. You may have as many of these at once as you like; all of them are read-only.
  • &mut T — a mutable (exclusive) borrow. You may have exactly one, and while it exists no other borrow of that value may exist — not even a shared one.

In one line: at any moment a value may have either one &mut or any number of &, never both. (Shared XOR mutable.) A second, quieter rule rides along: a reference may never outlive the data it points to — Rust simply will not compile a dangling reference. Making that second rule explicit in type signatures is the job of lifetimes, which you meet in Lesson 9.

This is how Rust eliminates data races and aliasing bugs at compile time, with no garbage collector. In C, nothing stops two mutable pointers to the same object, and the bugs that follow are yours to find. In Go, aliasing is free and a data race is caught — if you are lucky — at runtime by the race detector. Rust pulls that check forward into the compiler: a program that violates shared-XOR-mutable does not build.

Step 3 — Run it, and keep the tests green

The running tally has a natural home in the test suite too. Add one test for add_into alongside the existing three:

    #[test]
    fn counts_empty() {
        // ... unchanged ...
    }

    #[test]
    fn add_into_accumulates() {
        let mut total = (0, 0, 0);
        add_into(&mut total, (1, 2, 3));
        add_into(&mut total, (10, 20, 30));
        assert_eq!(total, (11, 22, 33));
    }

Then exercise the tool on more than one file and run the suite:

cargo run -- ../MISSION.md ../NOTES.md
cargo test

You should see a per-file line for each path plus a total line, and all four tests passing:

    19    199   1229 ../MISSION.md
    55   1223   8500 ../NOTES.md
    74   1422   9729 total

running 4 tests
test result: ok. 4 passed; 0 failed; 0 ignored; ...

(Your exact numbers will differ as the files grow — the point is the total row is the column-wise sum of the others.) The three Lesson 3 tests still pass unchanged: that is the safety net working. The refactor reshaped main substantially, and the tests confirm the counting itself never moved.

Outcome rwc now counts any number of files and totals them, and you have used both kinds of reference for real: a shared & to read a value without consuming it (&text, &paths) and an exclusive &mut to modify one in place (&mut total). The shared-XOR-mutable rule is the single idea the rest of Rust's safety story is built on.

Optional extensions

  1. Provoke a borrow error (a preview of Lesson 5). Inside the loop, try to take a second borrow of total while the &mut is conceptually in play — for instance add let peek = &total; on the line before add_into(&mut total, …) and then use peek after the call. Read the E0502 error ("cannot borrow total as mutable because it is also borrowed as immutable"). This is the rule from the concept box, enforced. Then remove it.
  2. Borrow instead of copy. Change add_into's second parameter to c: &(usize, usize, usize) and pass &(lines, words, chars). It still compiles — reading through a shared borrow is always allowed — and it shows the by-value-vs-by-reference choice is yours, not the compiler's, for a small Copy type.
  3. Per-directory guard. fs::read_to_string fails on a directory. Try cargo run -- . and confirm the per-file error message now names the offending path.

Check your understanding

While a &mut T borrow of a value is held, what else may borrow that value?

Right — &mut is exclusive: while it exists, no other borrow (shared or mutable) of that value may exist. That is the "mutable" half of shared-XOR-mutable.

A mutable borrow is exclusive. While a &mut is held, no other borrow of that value — shared or mutable — is allowed. Many & borrows are fine only when there is no &mut.

Why loop with for path in &paths rather than for path in paths?

Yes — &paths borrows the vector, so the loop only reads it and paths survives. We need that to call paths.len() for the total line afterwards.

It's about ownership, not speed or capability. for path in paths would move the vector into the loop and consume it; &paths borrows it so it's still alive afterwards (we use paths.len()).

Why is add_into's second argument passed by value (c: (usize, usize, usize)) with no &?

Correct — the tuple is a small Copy type, duplicated cheaply bit-for-bit, so there's no ownership to protect by borrowing. A String there would move, and you'd borrow it instead.

It's a Copy type: a tuple of integers duplicates cheaply, so passing it by value costs nothing and there's no move to avoid. You could borrow it (extension 2) — the choice is yours.
