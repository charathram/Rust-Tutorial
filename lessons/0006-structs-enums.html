Structs & enums

Give rwc's data real names. The anonymous (usize, usize, usize) tuple becomes a Counts struct, input gains a Source enum so rwc can read standard input — and, because the logic moves into a library crate, your examples can be runnable doc-tests.

Lesson goals

By the end of this lesson you will be able to:

  • Define a struct with named fields and give it methods in an impl block.
  • Use #[derive(...)] to get Debug, Default, Clone/Copy, and equality for free.
  • Model "one of several shapes" with an enum whose variants carry data, and read a variant back out with match.
  • Split a crate into a library (src/lib.rs) and a thin binary (src/main.rs) that uses it.
  • Write doc-tests — runnable examples in /// comments — and control them with hidden # lines and fence annotations (no_run, ignore, should_panic, compile_fail).

Why structs and enums, and why now

Through Lesson 5 the counts were a bare tuple, (usize, usize, usize). It worked, but every use site had to remember that .0 is lines, .1 is words, and .2 is characters — nothing in the code said so. A struct fixes that: it gives the three numbers names, so you write c.lines instead of c.0, and it can carry its own methods. An enum lets you say that input is either standard input or a named file and nothing else, which is what lets rwc read piped input the way real wc does.

Coming from C, a struct is the familiar record — but Rust's also owns its fields and can have methods. The enum goes further: it is a tagged union (a "sum type") where the compiler tracks which variant is active and a variant can hold data. C unions give you the overlapping storage but not the safety; Go has no direct equivalent and reaches for an interface or a spare boolean. Putting this logic into a library crate is also what makes doc-tests possible, the testing feature mentioned back in Lesson 3.

Step 1 — Start a library with a Counts struct

Until now everything lived in src/main.rs. Create a second file, src/lib.rs — cargo automatically treats it as a library crate named after the package (rwc). Put the counting logic there, built around a new Counts struct. Here is the file — from_text is written out as your model, and the second method, add, is left for you to fill in:

#[derive(Debug, Default, Clone, Copy, PartialEq, Eq)]
pub struct Counts {
    pub lines: usize,
    pub words: usize,
    pub bytes: usize,
}

impl Counts {
    pub fn from_text(text: &str) -> Counts {
        Counts {
            lines: text.lines().count(),
            words: text.split_whitespace().count(),
            bytes: text.len(),
        }
    }

    pub fn add(&mut self, other: Counts) {
        // TODO (your turn): add each field of other into self
    }
}

Your turn — implement add. It is the method form of Lesson 4's add_into: add each field of other onto self in place. The &mut self receiver is what lets you write back into the caller's value.

Show solution 
    pub fn add(&mut self, other: Counts) {
        self.lines += other.lines;
        self.words += other.words;
        self.bytes += other.bytes;
    }

Every piece, explained:

  • pub struct Counts { pub lines: usize, ... } — A struct defines a new type made of named fields. Counts bundles the three tallies into one value where c.lines says what c.0 never could. pub on the struct makes the type visible outside the library; pub on each field makes that field readable and writable from outside too (without it, the fields would be private to the library — fine for many designs, but here the binary needs to read them to print).
  • #[derive(Debug, Default, Clone, Copy, PartialEq, Eq)] — An attribute that asks the compiler to generate standard trait implementations so you don't write them by hand. Debug enables {:?} formatting (handy in tests and dbg!); Default gives Counts::default() — every field set to its own default, i.e. 0 — which is our zero-initialised total; Clone + Copy make Counts behave like the tuple did (cheap bit-for-bit duplication, no moves to worry about, because it is just three integers); PartialEq + Eq let you compare two Counts with == and use assert_eq! on them.
  • impl Counts { ... } — An implementation block: where a type's methods and associated functions live. Everything inside is attached to Counts.
  • pub fn from_text(text: &str) -> Counts — An associated function (no self parameter), called as Counts::from_text(...). It is the old counts() function, now returning a named Counts by writing each field — with one change from the tuple: the third count is now bytes (text.len()), what wc reports by default, rather than characters. The two are identical for ASCII and differ only on multi-byte UTF-8, which the Unicode test in Step 5 checks. The struct-literal syntax Counts { lines: ..., words: ..., bytes: ... } constructs a value by naming every field.
  • pub fn add(&mut self, other: Counts) — A method: the &mut self first parameter means it borrows the receiver mutably, so total.add(c) updates total in place. This replaces the free-standing add_into(&mut total, c) from Lesson 4 — same mutable-borrow idea, now written as total.add(c), which reads better and keeps the behaviour next to the data. self is just Rust's this; &mut self is shorthand for self: &mut Counts.

The binary won't compile against this yet — we rewire main in Step 3. First, the input side.

Step 2 — Model the input with a Source enum

So far rwc only reads files. Real wc also reads standard input when given no arguments (so you can pipe into it: cat file | wc). "A file path, or standard input" is a textbook case for an enum — a value that is exactly one of a fixed set of shapes. Add a Source enum below Counts, plus the two imports it needs at the top of the file — here is src/lib.rs now, with the new lines highlighted:

use std::fs;
use std::io::{self, Read};

#[derive(Debug, Default, Clone, Copy, PartialEq, Eq)]
pub struct Counts {
    // ... unchanged ...
}

impl Counts {
    // ... unchanged ...
}

pub enum Source {
    Stdin,
    File(String),
}

impl Source {
    pub fn from_args(args: Vec<String>) -> Vec<Source> {
        if args.is_empty() {
            vec![Source::Stdin]
        } else {
            args.into_iter().map(Source::File).collect()
        }
    }

    pub fn name(&self) -> &str {
        match self {
            Source::Stdin => "-",
            Source::File(path) => path,
        }
    }

    pub fn read(&self) -> io::Result<String> {
        // TODO (your turn): match self — Stdin reads io::stdin() into a
        // String; File(path) uses fs::read_to_string(path). Let ? bubble.
    }
}

Your turn — implement read. It has the same match shape as name just above: for Source::File(path) reuse fs::read_to_string from Lesson 2; for Source::Stdin read io::stdin() into a String with read_to_string, letting ? bubble any error.

Show solution 
    pub fn read(&self) -> io::Result<String> {
        match self {
            Source::Stdin => {
                let mut buf = String::new();
                io::stdin().read_to_string(&mut buf)?;
                Ok(buf)
            }
            Source::File(path) => fs::read_to_string(path),
        }
    }

What is new here:

  • pub enum Source { Stdin, File(String) } — An enum declares a type with a fixed list of variants. A Source is either Stdin (a plain variant, no data) or File(String) (a variant that carries a String, the path). Crucially it can only ever be one of them — the compiler stores a hidden tag and won't let you read the path out of a Stdin. This is the "sum type" idea: unlike a struct, which holds all its fields at once, an enum holds exactly one variant at a time.
  • use std::fs; and use std::io::{self, Read}; — Imports. io::{self, Read} brings in both the io module itself (so we can write io::stdin() and io::Result) and the Read trait; the trait must be in scope for its .read_to_string() method to be callable on stdin. (Traits get their own lessons in Module 4; for now just know a method can come from a trait you have to import.)
  • from_args — Turns the argument list into a Vec<Source>. No arguments ⇒ a single Source::Stdin, matching wc's behaviour. Otherwise each path string becomes a Source::File. The expression .map(Source::File) uses a tuple-variant's name as a function: Source::File is, in effect, a function String -> Source, so mapping it over the paths wraps each one. (Writing .map(|p| Source::File(p)) means the same thing; clippy will nudge you to the shorter form.)
  • name(&self) -> &str — A method taking &self (a shared borrow — it only reads). It returns the label for the output column. The match inspects which variant self is; for File(path) it binds the inner string to path and returns it. match is Rust's version of switch, with more capability (exhaustiveness checking, richer patterns); Lesson 7 covers it in full. Here it is simply the way to ask an enum "which variant are you, and what's inside?".
  • read(&self) -> io::Result<String> — Reads the whole source into a String. For Stdin it reads from io::stdin() into a buffer (the ? bubbles up any IO error, exactly as in Lesson 2); for File(path) it calls the same fs::read_to_string you already know. Both arms produce an io::Result<String>, so the whole match does.
Concept · struct vs enum

Both define new types; the difference is "and" versus "or":

  • A struct is a record of fields held together — a Counts has lines and words and bytes, all at once. (A "product type".)
  • An enum is a choice between variants held one at a time — a Source is Stdin or File, never both. (A "sum type".) Variants may carry their own data of different shapes.

You have already been using enums: Option<T> (Some/None) and Result<T, E> (Ok/Err) are both enums from the standard library. Defining Source is the same machinery, now in your own code.

Step 3 — Make main a thin client of the library

With the logic in the library, the binary's job shrinks to: parse args into sources, read and count each, print, total. Replace the contents of src/main.rs entirely:

use rwc::{Counts, Source};
use std::env;
use std::error::Error;

fn main() -> Result<(), Box<dyn Error>> {
    let args: Vec<String> = env::args().skip(1).collect();
    let sources = Source::from_args(args);

    let mut total = Counts::default();
    for source in &sources {
        let text = source.read().map_err(|e| format!("{}: {}", source.name(), e))?;
        let c = Counts::from_text(&text);
        println!("{:>6} {:>6} {:>6} {}", c.lines, c.words, c.bytes, source.name());
        total.add(c);
    }

    if sources.len() > 1 {
        println!("{:>6} {:>6} {:>6} total", total.lines, total.words, total.bytes);
    }
    Ok(())
}

What each new line does:

  • use rwc::{Counts, Source}; — The binary is a separate crate from the library, so it imports the library by package name, rwc, exactly as it would any external crate. This is the line that connects the two halves; only pub items (which is why we marked them pub in Steps 1–2) are reachable here.
  • let sources = Source::from_args(args); — One call turns the raw arguments into the list of sources. The empty-args case (read stdin) is now handled inside the library, so main doesn't special-case it.
  • let mut total = Counts::default(); — The zero total, now a named value from the derived Default instead of (0, 0, 0).
  • source.read().map_err(|e| format!("{}: {}", source.name(), e))? — Reading is now a method on the source, so the same line works whether it is a file or stdin. The map_err prefixes the failing source's name to the error (for stdin that name is -), and ? bubbles the first failure out of main as before.
  • total.add(c); — Method-call form of the accumulator. c is the per-source Counts; total.add(c) folds it in.

Everything now compiles again. The behaviour for files is identical to Lesson 5; the new ability is that rwc with no arguments reads standard input.

Step 4 — Document the library with doc-tests

This is the testing topic flagged back in Lesson 3. A doc-test is an example written inside a /// documentation comment, fenced in triple-backticks. cargo test compiles and runs each one — so your documentation can never silently drift out of date with the code. The catch noted in Lesson 3: doc-tests run only for library crates. That is precisely why we created src/lib.rs in this lesson — the examples below would be invisible to cargo test if this logic had stayed in main.rs.

Add /// comments with examples above the items in src/lib.rs. Here is Counts and its two methods, doc-comments highlighted:

/// A tally of lines, words, and bytes for one piece of text.

    /// assert_eq!(c.words, 3);
    /// assert_eq!(c.lines, 1);
    /// ```
    pub fn from_text(text: &str) -> Counts {
        // ... unchanged ...
    }

    /// Add another tally into this one, field by field.
    ///
    /// ```
    /// # use rwc::Counts;
    /// let mut total = Counts::default();
    /// total.add(Counts::from_text("a b\n"));
    /// total.add(Counts::from_text("c d\n"));
    /// assert_eq!(total.lines, 2);
    /// assert_eq!(total.words, 4);
    /// ```
    pub fn add(&mut self, other: Counts) {
        // ... unchanged ...
    }
}

The first example sits on the struct itself; the other two sit on individual methods. A doc comment — and any example inside it — can attach to any item (a module, type, method, or field), and cargo test runs every fenced example regardless of where it lives. The struct-level one is a how to use this type overview, so it builds a Counts with from_text and checks the fields; each method example focuses on that one method. (When you run the suite in Step 5 you'll see them listed as Counts, Counts::from_text, and Counts::add — the name is just the item the example sits on, not what it tests.)

The two doc-test mechanics on display:

  • The fenced block runs as a test. The triple-backtick block is compiled into a tiny program and executed; the assert_eq! lines are the actual assertions. If from_text ever returned the wrong count, this test would fail just like a unit test — so the example in your docs is guaranteed to be correct and current.
  • # hides a line from the rendered docs but still runs it. In the first example we show use rwc::Counts; so a reader sees the import; in the later two we write # use rwc::Counts; — the leading # means "compile and run this line, but don't display it." It is how you keep examples uncluttered (boilerplate imports, setup) while keeping them runnable. (Note: a String with a literal newline in source is written "a b\n" — the same escape you'd use in C.)

Now document Source. The interesting one is read, whose example must not actually run — reading real standard input would hang the test suite waiting for keyboard input. That is what a fence annotation is for:

impl Source {
    /// Read the entire contents of this source into a `String`.
    ///
    /// ```no_run
    /// # use rwc::Source;
    /// let text = Source::Stdin.read()?;
    /// println!("{} bytes", text.len());
    /// # Ok::<(), std::io::Error>(())
    /// ```
    pub fn read(&self) -> io::Result<String> {
        // ... unchanged ...
    }
}
  • ```no_run — The example is still compiled (so the code is type-checked and can't rot), but it is not executed. Perfect for code that does real IO, networking, or anything you don't want running inside cargo test.
  • The hidden # Ok::<(), std::io::Error>(()) line — Because the example uses ?, the generated test function has to return a Result. This hidden final line supplies the matching Ok(()) (with the error type spelled out via the ::<...> "turbofish") so the snippet type-checks — without the reader seeing the plumbing.

The doc-test fence annotations

The word after the opening backticks controls how the example is treated. The four you will reach for:

  • ``` (none) — compile and run; the default. Assertions inside are real test failures.
  • ```no_run — compile but don't run. For examples that do IO, networking, or open windows — checked for correctness, never executed.
  • ```ignore — neither compile nor run. The escape hatch for pseudo-code or a fragment that can't stand alone. Use sparingly: an ignored example is unverified, so it can rot.
  • ```should_panic — compile and run, and the test passes only if the example panics. Use it to document that a function rejects bad input (e.g. an example that triggers an unwrap() on None).
  • ```compile_fail — the test passes only if the example fails to compile. Niche but powerful: it lets you prove in your docs that a misuse is rejected by the type system — for instance, that you can't read a moved value.

Reach for the default whenever you can: a runnable example with assertions is documentation that cargo test verifies, so it can't drift out of date with the code.

Outcome rwc is now built from named types: a Counts struct whose named fields say what the tuple positions never did and whose derives give you Default, equality, and copy semantics for free, and a Source enum that models file vs stdin, which is what makes reading piped input work. The logic lives in a library crate, which let you write runnable doc-tests — examples that cargo test compiles and runs — alongside the unit tests. One cargo test now runs unit tests and doc-tests, and clippy is clean.

Optional extensions

  1. Implement Display for Counts. Instead of formatting the columns in main, give Counts a std::fmt::Display implementation so you can write println!("{c} {}", source.name()). (This is a first taste of implementing a trait by hand — the subject of Module 4.)
  2. Add a should_panic doc-test. Write a small associated function that unwrap()s something that can be None, and document it with a ```should_panic example proving it panics on the bad input. Run cargo test and watch it pass because it panicked.
  3. Try a compile_fail doc-test. Add a ```compile_fail example that takes a &mut of an immutable binding (the E0596 from Lesson 5). The doc-test passes precisely because that code does not compile — your documentation now proves a misuse is rejected.
  4. Support - as an explicit stdin argument. Real wc treats a lone - on the command line as "read stdin here." Extend from_args to map the string "-" to Source::Stdin. (You'll want match or an if on the path — a natural lead-in to Lesson 7.)
